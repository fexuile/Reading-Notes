# PrefillOnly: An Inference Engine for Prefill-only Workloads in Large Language Model Applications

## 基本信息

- 论文：PrefillOnly: An Inference Engine for Prefill-only Workloads in Large Language Model Applications
- 作者：Kuntai Du, Bowen Wang, Chen Zhang, Yiming Cheng, Qing Lan, Hejian Sang, Yihua Cheng, Jiayi Yao, Xiaoxuan Liu, Yifan Qiao, Ion Stoica, Junchen Jiang
- 会议：SOSP 2025
- 主题：LLM inference engine、prefill-only workload、KV cache 管理、调度、prefix caching

## 背景与问题

传统 LLM 推理系统主要面向生成式任务，例如聊天、代码生成和摘要。这类任务通常包含两个阶段：先通过 prefill 处理完整输入 prompt，再通过多轮 decode 逐 token 生成输出。因此，现有推理引擎通常会为所有层、所有历史 token 保存 KV cache，以便后续 decode 复用。

本文观察到，LLM 正越来越多地用于推荐、信用审核、数据标注等判别式任务。在这些任务中，系统通常不需要模型生成一段长文本，而只需要模型对一个候选 token 或一组候选 token 给出概率分布。例如推荐系统只需要 `P(Yes)` 和 `P(No)`，并将 `P(Yes)` 作为推荐分数。论文将这类请求称为 *prefill-only request*，将对应负载称为 *prefill-only workload*。

这类负载有两个重要特点：

- 每个请求只需要生成一个 token，因此不需要多轮 decode。
- 输入通常很长，例如用户画像、浏览历史、信用记录或长文档。

现有 LLM 推理引擎没有充分利用这两个特点。它们仍然假设请求可能继续 decode，因此保存完整 KV cache，导致长输入时显存压力很大；同时，它们通常使用不感知作业完成时间的调度策略，无法利用 prefill-only 请求完成时间更可预测这一性质。

## 核心思想

PrefillOnly 的核心观点是：如果请求只需要 prefill，那么为未来 decode 保存完整 KV cache 是不必要的。系统可以重新设计显存管理和调度策略，以支持更长输入并提升吞吐。

论文提出的系统 PrefillOnly 主要包含三项技术：

1. **Hybrid Prefilling**
2. **Suffix KV Cache Discarding / Offloading**
3. **Shortest Prefill First with Continuous Prefill Time Estimation**

## Hybrid Prefilling

直觉上，prefill-only 请求不需要保存所有层的 KV cache。当前层 attention 计算完后，该层 K/V 对当前请求通常就不再有用，因为之后不会有 decode token 回来访问这些历史 K/V。

但论文指出，简单地“只保留一层 KV cache”并不能显著提升最大输入长度。原因是 LLM forward 过程中还有大量临时中间张量，尤其是 MLP/linear 层中的 intermediate tensors。这些张量可能比单层 KV cache 更大，仍然会造成显存峰值。

为此，PrefillOnly 使用 hybrid prefilling：

- attention 层不切块，保持完整上下文计算，避免降低 attention kernel 性能；
- 非 attention 层，特别是 linear/MLP 层，按 chunk 处理，从而降低中间张量峰值显存；
- 通过 `torch.compile` 修改计算图，将连续 linear 操作组合成 virtual layer，再对该 virtual layer 分块执行；
- 进一步使用输出预分配和 in-place 计算减少额外显存开销。

这个设计的关键是，它减少了 MLP 中间张量的显存峰值，同时不破坏 attention 层对完整上下文的处理。

## Suffix KV Cache Discarding / Offloading

PrefillOnly 还需要兼顾 prefix caching。许多 prefill-only 场景中，不同请求可能共享很长的前缀，例如同一用户画像加不同候选帖子。保留前缀 KV cache 可以让后续请求复用已有计算，减少延迟和计算量。

因此，PrefillOnly 的策略不是丢弃所有 KV cache，而是：

- 尽量保留 prefix token 的 KV cache，用于未来 prefix caching；
- 对超出 GPU 缓存容量的 suffix token KV cache 进行丢弃或 offload；
- 当前实现主要是 discard，论文指出也可以结合 LMCache 或 Mooncake Store 等系统进行 CPU/外部存储 offload。

Hybrid prefilling 是 suffix KV cache discarding/offloading 的基础。因为 attention 层保持不切块，prefill 可以在一次完整 forward 中完成；系统可以在每层计算过程中丢弃或卸载不需要长期保留的 suffix KV cache，而不需要像 chunked prefilling 那样反复 reload 或 regenerate 前面 chunk 的 K/V。

## 调度：Shortest Prefill First with Continuous Estimation

Prefill-only 请求的输出长度固定为 1，因此比普通生成式请求更容易估计完成时间。论文利用这一点，采用类似 shortest job first 的策略，即优先处理预计 prefill 时间最短的请求。

但 prefix caching 会让请求的实际 prefill 时间动态变化。如果某个请求的前缀已经在 cache 中，那么它只需要计算未命中的 suffix，执行时间会显著缩短。反之，如果 cache 被驱逐，请求时间又会变长。

因此，PrefillOnly 提出 continuous prefill time estimation：

- 每次调度前，重新估计等待队列中所有请求的 prefill 时间；
- 估计时考虑输入长度和 prefix cache 命中 token 数；
- 使用 cache-miss token 数量作为 prefill 时间的近似指标；
- 加入 `-lambda * queue_time` 项，避免短请求持续插队造成长请求饥饿。

这种调度同时实现了两个目标：降低平均延迟，并提高 prefix cache 命中率。

## 实验设置与结果

论文在 vLLM 基础上实现 PrefillOnly，核心代码约 4.6k 行 Python。实验覆盖：

- 4 种硬件设置：NVIDIA L4、A100、H100 PCIe、H100 NVLink；
- 3 个 LLM 模型；
- 2 类模拟负载：post recommendation 和 credit verification；
- baselines 包括 PagedAttention、chunked prefilling、pipeline parallelism 和 tensor parallelism。

主要结果如下：

- 在不增加平均延迟和 P99 延迟的情况下，PrefillOnly 可以支持约 1.4 到 4.0 倍更高 QPS。
- 相比非并行 baseline，PrefillOnly 可将最大输入长度提高最多约 5 倍。
- Hybrid prefilling 显著提升 maximum input length，同时避免 chunked prefilling 对 attention kernel 和吞吐的损害。
- 在高 QPS 下，PrefillOnly 的吞吐和延迟优势更明显；在低 QPS 下，并行方案有时有更低单请求延迟，但由于通信和同步开销，扩展到高负载时吞吐较差。

实验中的 routing 也值得注意。对于 PrefillOnly 和非并行 baseline，系统在每张 GPU 上启动一个完整 inference engine 实例，并使用 user-id-based routing：同一用户的请求固定发送到同一实例，以提高 prefix cache 复用率；不同用户按 round-robin 分配以平衡负载。

## 局限性

论文也讨论了几个限制：

- 当前实现主要丢弃 suffix KV cache，而不是完整实现 offloading，因此可能损失未来请求复用 suffix cache 的机会。
- 当 workload 具有大量长前缀共享，并且 prefix cache 空间本身成为瓶颈时，tensor parallel 或 pipeline parallel 这类并行化方案可能更合适，因为它们可以把 KV cache 分摊到多张 GPU 上。
- PrefillOnly 的调度主要在单个 inference engine 内部完成。大规模部署中，可能需要在 router 层做全局 shortest prefill first routing。
- 论文主要优化吞吐，对极致单请求延迟的优化还有空间。

## 阅读过程中的问题与理解

阅读这篇论文时，我一开始的主要困惑集中在 LLM inference 的基本 workflow 上。后来我把普通生成式推理理解为两个阶段：**prefill** 是一次性处理完整 prompt，建立每一层 attention 的 K/V，并得到第一个输出 token 的概率分布；**decode** 是在此之后逐 token 生成，每一步只输入新生成的 token，同时复用历史 KV cache。这个区分解释了为什么 prefill-only workload 特殊：它只需要 prefill 结束时的一个 token 概率分布，不需要后面的 decode 循环。

我也进一步澄清了 KV cache 的含义。KV cache 存储的不是训练阶段的东西，也不是模型参数，而是当前推理请求中，由训练好的 `W_K` 和 `W_V` 根据输入 hidden states 计算出来的 K/V 激活值。训练学到的是 `W_Q`、`W_K`、`W_V` 这些投影矩阵；推理时每个 token、每一层都会临时算出自己的 Q/K/V。KV cache 保存的是 K 和 V，因为未来 decode token 会用自己的 Q 去 attend 历史 K/V。

另一个关键问题是：为什么会有很多层 attention，以及 PrefillOnly 所说的“一层 KV cache”中的“层”指什么。这里的层指 Transformer layer 或 Transformer block 中的 attention 层。普通生成式推理必须保存所有层的 KV cache，因为新 token 在 decode 时会依次经过所有 Transformer layers，并且在第 `l` 层需要访问历史 token 在同一第 `l` 层的 K/V。PrefillOnly 只做 prefill，因此某一层 attention 算完后，该层 K/V 对当前请求通常不再需要，所以理论上只需要保留当前正在计算的一层 K/V。

我还把 prefix caching 和 suffix KV cache discarding/offloading 的关系理清了。Prefix caching 是指多个请求共享同一个 prompt 前缀时，系统保留这个前缀的 KV cache，让后续请求复用。例如同一个用户画像加不同候选帖子时，用户画像就是共享 prefix。PrefillOnly 不能简单丢弃全部 KV cache，因为这样会损失 prefix caching 的收益；它选择尽量保留 prefix KV cache，并丢弃或卸载复用价值较低的 suffix KV cache。

在理解 Hybrid Prefilling 和 Suffix KV Cache Offloading 的关系时，我的结论是：Hybrid Prefilling 是计算方式上的改造，Suffix KV Cache Offloading 是缓存管理策略。前者通过让 attention 层不切块、非 attention 层切块，降低 MLP 中间张量显存峰值，并使系统可以在一次完整 prefill 中安全丢弃或卸载 suffix KV cache。没有 hybrid prefilling，普通 chunked prefilling 可能需要反复使用前面 chunk 的 K/V，suffix cache 的丢弃/offload 就不容易做得高效。

最后，7.1 中的 routing 也帮助我理解了实验设置。对于非并行方案，系统在每张 GPU 上跑一个完整 inference engine 实例，然后按 user-id 路由：同一个用户的请求固定到同一个实例，以提高 prefix cache 命中率；不同用户通过 round-robin 分配以平衡负载。相比之下，tensor parallel 或 pipeline parallel 是一个模型实例跨多张 GPU，因此 KV cache 和模型参数被分摊到多张 GPU 上，有更多 prefix KV cache 空间，但代价是通信和同步开销。

## 我的理解

这篇论文的重要性在于，它指出 LLM serving 系统不应该只围绕生成式聊天 workload 设计。推荐、审核、分类、embedding 等应用中的 LLM 调用，经常只需要一次 prefill 和一个 token 的概率分布。这类 workload 的系统瓶颈和普通生成式 workload 不同，因此需要不同的显存管理和调度策略。

PrefillOnly 的设计逻辑很清晰：

- 既然没有多轮 decode，就不必为当前请求保留完整 KV cache；
- 既然输入很长，就必须控制 MLP 中间张量和 KV cache 两类显存开销；
- 既然输出长度固定，就可以进行更激进的 JCT-aware scheduling；
- 既然很多请求共享前缀，就不能简单丢弃所有 KV cache，而要保留最有复用价值的 prefix。

整体来看，PrefillOnly 是一篇典型的系统论文：它从新 workload 的特性出发，重新审视已有系统假设，并围绕显存、缓存和调度三个方面做专门优化。

# MCM-GPU: Multi-Chip-Module GPUs for Continued Performance Scalability

## 基本信息

- 论文：MCM-GPU: Multi-Chip-Module GPUs for Continued Performance Scalability
- 作者：Akhil Arunkumar, Evgeny Bolotin, Benjamin Cho, Ugljesa Milic, Eiman Ebrahimi, Oreste Villa, Aamer Jaleel, Carole-Jean Wu, David Nellans
- 会议：ISCA 2017
- 主题：GPU architecture、multi-chip module、chiplet GPU、NUMA、on-package interconnect、CTA scheduling、cache hierarchy

## 背景与问题

传统 GPU 性能增长很大程度上依赖晶体管缩放：随着工艺进步，单个 GPU die 可以集成更多 Streaming Multiprocessors（SM）、更大 cache 和更高内存带宽。但论文指出，这条路径会遇到几个限制：

- Moore's Law 放缓后，单位面积可集成晶体管数不再快速增长；
- 光刻 reticle size 限制了单个 die 的最大面积；
- 大 die 良率低，制造成本高；
- 即使应用有足够并行度，单芯片 GPU 也无法无限增加 SM 数量。

multi-GPU 是一种自然替代方案，但它也有明显问题。多个完整 GPU 通常通过板级或系统级互连协作，带宽更低、延迟更高、能耗更大，并且常常需要程序员或 runtime 显式处理任务划分、数据放置、同步和负载均衡。

本文提出的核心问题是：**能否在不制造超大 monolithic GPU die 的情况下，继续扩展单个逻辑 GPU 的性能？**

## 核心思想

论文提出 *MCM-GPU*，即把多个 GPU Module（GPM）集成到同一个 package 中，通过封装内互连连接，并对 OS 和程序员暴露成一个逻辑 GPU。

它的位置介于 monolithic GPU 和 multi-GPU 之间：

- monolithic GPU：所有计算和内存系统都在一个大 die 上；
- multi-GPU：多个完整 GPU 在 board/system 层级互连；
- MCM-GPU：多个较小 GPM/chiplet 在同一个 package 内互连，共同组成一个逻辑 GPU。

论文评估的主要设计是一个 256 SM 的 MCM-GPU，由 4 个 64 SM 的 GPM 组成。每个 GPM 包含一部分 SM、L1 cache、crossbar、L2 slice 和本地 DRAM partition。所有 GPM 的资源共同形成一个统一地址空间和一个逻辑 GPU。

## MCM-GPU 的组织方式

MCM-GPU 中的 L2 cache 容易引起混淆。更准确的理解是：

> L2 是逻辑上 shared、物理上 distributed/sliced 的。

也就是说，总 L2 由多个 GPM 内的 L2 slice 组成。所有 SM 都可以访问任意 L2 slice，但每个 physical address 只归属于一个 home GPM 的 L2 slice 和 DRAM partition。

例如：

```text
GPM0 的 SM 访问地址 A

如果 A 映射到 GPM0：
    GPM0 SM -> GPM0 L2 slice -> GPM0 DRAM

如果 A 映射到 GPM2：
    GPM0 SM -> inter-GPM link -> GPM2 L2 slice -> GPM2 DRAM
```

因此，baseline MCM-GPU 中每个 L2 slice 只缓存本地 DRAM partition 对应的数据。同一条 cache line 不会同时存在于多个 L2 slice 中，所以 L2 slice 之间不需要复杂硬件一致性协议。

## On-Package Bandwidth Considerations

这一节的核心问题是：**GPM 之间的封装内互连带宽要多大，才能不拖累整体 GPU 性能？**

论文用 4-GPM 系统进行估算。设每个 GPM 的本地 DRAM partition 带宽为 `b`，那么总 DRAM 带宽为 `4b`。在论文的配置中：

```text
总 DRAM bandwidth = 3 TB/s
每个 GPM 本地 DRAM bandwidth = b = 768 GB/s
```

如果地址在所有 GPM 的 memory partitions 上细粒度均匀分布，那么一个 GPM 的访存请求中，只有一部分访问本地 partition，很多请求会访问远端 partition。为了让所有 GPM 都能充分利用总 DRAM 带宽，inter-GPM bandwidth 理想上需要接近总 DRAM bandwidth，也就是约 `3 TB/s`。

但是这样的封装内互连成本、布线复杂度和功耗都较高。论文基于 NVIDIA 的 Ground-Referenced Signaling（GRS）技术，认为 `768 GB/s` 是更现实、低成本、低功耗的设计点。GRS 单线速率约为 `20 Gb/s`，因此 `768 GB/s` 级别带宽需要数百条并行高速信号线在 package 内连接相邻 GPM。

这也解释了后文优化的动机：baseline 选择的 `768 GB/s` inter-GPM link 低于理想需求，所以必须通过架构机制减少跨 GPM 流量。

## 三项优化机制

论文提出三个 locality-aware 优化，目标都是减少 remote access 和 inter-GPM bandwidth pressure。

### Remote-Only L1.5 Cache

L1.5 cache 是每个 GPM 内新增的一层 GPM-side cache，位于 SM 私有 L1 和远端 L2/DRAM 之间。它主要缓存本 GPM 访问过的远端数据。

baseline 中，如果 GPM0 访问 GPM2 的数据，后续再次访问时仍可能需要跨 GPM。加入 L1.5 后，第一次远端访问会把数据带回 GPM0 的 L1.5，后续访问可以在本地命中，从而减少 inter-GPM traffic 和远端访问延迟。

论文评估了 6 种 L1.5 配置：

| 配置 | 含义 |
|---|---|
| `8 MB L1.5` | 总 L1.5 为 8 MB，同时缓存本地和远端数据 |
| `8 MB Remote-Only L1.5` | 总 L1.5 为 8 MB，只缓存远端数据 |
| `16 MB L1.5` | 总 L1.5 为 16 MB，同时缓存本地和远端数据 |
| `16 MB Remote-Only L1.5` | 总 L1.5 为 16 MB，只缓存远端数据 |
| `32 MB L1.5` | 总 L1.5 为 32 MB，同时缓存本地和远端数据，额外增加 cache 容量 |
| `32 MB Remote-Only L1.5` | 总 L1.5 为 32 MB，只缓存远端数据，额外增加 cache 容量 |

其中 `8 MB` 和 `16 MB` 是 iso-transistor 配置，主要通过把原本 L2 的一部分容量挪到 L1.5 实现。`32 MB` 则额外增加 cache 容量，不是公平面积比较。

实验显示，remote-only 策略更符合 L1.5 的目标，因为本地数据已经由本地 L2 处理，L1.5 应优先保留代价更高的远端数据。单独使用 L1.5 时，`16 MB Remote-Only L1.5` 是两个 iso-transistor 配置中效果最好的设计。

### Distributed CTA Scheduling

CTA（Cooperative Thread Array）基本等同于 CUDA 中的 thread block。CTA 是 GPU 调度到 SM 上执行的基本单位。

baseline 的 centralized CTA scheduling 会根据 SM 可用情况全局分配 CTA，可能导致相邻 CTA 被分散到不同 GPM 中。例如：

```text
CTA0 -> GPM0
CTA1 -> GPM1
CTA2 -> GPM2
CTA3 -> GPM3
```

但相邻 CTA 往往访问相邻或重叠数据。如果它们被分散到不同 GPM，就会破坏 cache locality 并增加跨 GPM 访问。

Distributed CTA scheduling 将连续 CTA 分组分配到同一个 GPM，例如：

```text
CTA0, CTA1, CTA2, CTA3 -> GPM0
CTA4, CTA5, CTA6, CTA7 -> GPM1
```

这样可以提高 L1.5 cache 命中率，并减少 inter-GPM traffic。论文也指出，这种固定粗粒度分组可能造成负载不均，因此未来可以考虑动态选择 CTA group size。

### First-Touch Page Placement

First-touch（FT）策略用于页面放置。规则是：一个 memory page 第一次被哪个 GPM 访问，就被映射到哪个 GPM 的 local memory partition。

例如：

```text
CTA-X 在 GPM0 上第一次访问 page P0
=> P0 被放到 GPM0 的 memory partition

之后 GPM1 访问 P0
=> P0 仍在 GPM0
=> GPM1 需要远程访问
```

因此，FT 是 first-touch placement，不是动态 page migration。页面第一次放置后通常不再移动。

FT 和 distributed CTA scheduling 有很强的协同作用。很多 GPU 应用会反复 launch 相同 kernel，具有相同 index 的 CTA 在多次 kernel invocation 中可能访问相同页面。如果 distributed scheduling 让这些 CTA 稳定落在同一个 GPM，FT 就能让相关页面也稳定放在该 GPM 的本地 memory partition，从而跨 kernel 保持局部性。

## 实验设置与结果

论文使用 NVIDIA 内部模拟器评估一个类似 Pascal、但规模扩展到 256 SM 的 GPU。baseline 配置包括：

- 4 个 GPM；
- 总共 256 SM；
- 每个 SM 128 KB L1 data cache；
- 总 L2 cache 16 MB；
- inter-GPM ring interconnect，每条 link 768 GB/s，每 hop 32 cycles；
- 总 DRAM bandwidth 3 TB/s；
- DRAM latency 100 ns。

benchmark 共 48 个，来自 CORAL、Lonestar、Rodinia 和 NVIDIA 内部 CUDA workloads。论文将 workload 分为 high-parallelism 和 limited-parallelism，并进一步把 high-parallelism 分为 memory-intensive 和 compute-intensive。

主要结果：

- baseline MCM-GPU 在 inter-GPM bandwidth 低于 3 TB/s 时会明显受 NUMA 效应影响，memory-intensive workload 最敏感。
- 单独加入 remote-only L1.5 cache 可带来约 5.2% 平均性能提升，并显著降低 inter-GPM bandwidth。
- L1.5 cache + distributed CTA scheduling 对 memory-intensive workloads 平均提升约 23.4%。
- 加入 FT 后，最佳组合变为 `8 MB Remote-Only L1.5 + 8 MB L2 + Distributed Scheduling + First Touch`，因为 FT 使很多访问变成本地访问，此时保留足够 L2 容量更重要。
- 三项机制组合后，相比 baseline MCM-GPU 平均性能提升 22.8%，inter-GPM bandwidth 降低 5 倍。
- 优化后的 MCM-GPU 比论文假设的最大可制造 128 SM monolithic GPU 快 45.5%。
- 优化后的 MCM-GPU 比同等资源 multi-GPU 快 26.8%，并且距离不可制造的理想 256 SM monolithic GPU 性能差距在 10% 以内。

## 局限性

这篇论文也有一些限制：

- 结果基于 NVIDIA 内部模拟器和前瞻性硬件假设，不是实物芯片测量结果。
- `768 GB/s` inter-GPM bandwidth 基于 GRS 技术能力估算，但论文没有制造真实 MCM-GPU 验证。
- inter-GPM topology 主要采用 ring/mesh 的高层抽象，没有深入探索拓扑、routing 和拥塞控制。
- distributed CTA scheduling 的粒度固定，可能导致负载不均。
- FT page placement 不包含动态页面迁移，因此如果 first touch 与后续主要访问者不一致，仍可能产生远端访问。
- L1.5 cache 采用 write-through 以适配 GPU 软件一致性模型，这可能对写密集 workload 产生额外 DRAM traffic。

## 阅读过程中的问题与理解

我一开始主要困惑在 chiplet、MCM 和 multi-GPU 的关系。后来理清为：chiplet 是组成单位，即较小的 die；MCM 是把多个 die/chiplet 放进同一个 package 的封装方式；multi-GPU 则是多个完整 GPU 在板级或系统级互连。MCM-GPU 的目标不是暴露多个 GPU，而是把多个 GPM 组合成一个逻辑 GPU。

另一个关键概念是 package。这里的 package 不是软件包，而是芯片封装层级，介于单个 die 和 PCB board 之间。MCM-GPU 利用 package-level interconnect，让 GPM 间通信比 board-level multi-GPU 更近、更宽、更省电。

我也进一步澄清了 L2 cache 的组织方式。Section 4 中说 shared L2，是指 L2 对所有 SM 逻辑共享；但物理实现上，L2 被切成多个 slice 分布在各个 GPM 内。每个地址只对应一个 home L2 slice 和 DRAM partition，所以 L2 是 distributed shared cache，而不是一个集中式 L2。

L1.5 cache 的角色也需要和 L2 区分。L2 是 memory-side cache，负责某个 GPM 本地 DRAM partition 的数据；L1.5 是 GPM-side cache，主要缓存本 GPM 曾经访问过的远端数据。它不是全局共享 cache，而是减少远端访问的局部副本缓存。

对于 on-package bandwidth，我的理解是：论文不是说 `768 GB/s` 已经在产品中实现，而是把它作为基于 GRS 技术趋势的可行设计点。理论上，为了完全利用 3 TB/s 总 DRAM bandwidth，inter-GPM bandwidth 需要接近 3 TB/s；但这样的互连太贵。因此作者选择较现实的 768 GB/s，然后用 cache、CTA scheduling 和 page placement 降低实际跨 GPM 流量。

## 我的理解

这篇论文的重要性在于，它较早地把 chiplet/MCM 思路系统性引入 GPU 性能扩展问题。它的核心洞察不是简单地“把多个小 GPU 拼起来”，而是指出这样的系统天然会成为 NUMA GPU，因此必须围绕 locality 重新设计 cache、调度和页面放置。

MCM-GPU 的设计逻辑可以概括为：

- 用多个较小 GPM 避免超大 monolithic die 的制造限制；
- 用 package-level interconnect 避免 board-level multi-GPU 的高延迟、高能耗和编程复杂度；
- 用统一逻辑 GPU 把复杂性尽量隐藏在硬件和驱动中；
- 用 L1.5 cache、distributed CTA scheduling 和 FT page placement 缓解 NUMA 带来的远端访问开销。

整体来看，MCM-GPU 是一篇典型的体系结构论文：它先指出未来单芯片扩展的物理限制，再提出一个新的封装级组织方式，最后围绕该组织方式产生的新瓶颈设计配套机制。它对后来的 chiplet GPU 和先进封装 GPU 设计有很强的启发意义。

# LegoOS: A Disseminated, Distributed OS for Hardware Resource Disaggregation

## 基本信息

- 论文：LegoOS: A Disseminated, Distributed OS for Hardware Resource Disaggregation
- 作者：Yizhou Shan, Yutong Huang, Yilun Chen, Yiying Zhang
- 会议：OSDI 2018
- 主题：resource disaggregation、distributed OS、splitkernel、remote memory、RDMA、datacenter OS

## 背景与问题

传统数据中心长期以 monolithic server 作为部署、运行和故障隔离单位。一台服务器内部包含 CPU、DRAM、磁盘、NIC、电源、主板等组件，应用的 CPU 和内存通常必须来自同一台物理机器。论文认为这种 server-centric 架构在现代数据中心中遇到四类问题。

第一是资源利用率低。CPU 和内存被绑定在同一台机器内，调度器必须把作业装入某台服务器的资源组合中。即使整个集群还有大量空闲 CPU 或内存，只要它们没有出现在同一台服务器上，作业仍然可能无法放置。论文用 Google 和 Alibaba 的生产集群 trace 说明，CPU 和内存平均利用率都只有约一半。

第二是硬件弹性差。一旦硬件被封装进服务器，后续添加、移除或重新组合某类资源都很困难。数据中心需要提前规划服务器配置，但应用需求变化很快，导致配置容易失配。

第三是故障域太粗。服务器内部任意关键组件故障，例如内存控制器、CPU、主板或电源故障，都可能导致整台服务器不可用。即使其他组件仍然正常，也会跟着失效。

第四是对异构硬件支持不好。现代数据中心越来越多地使用 GPU、TPU、DPU、FPGA、NVM 和 NVMe SSD 等专用硬件。传统服务器把硬件设备、主板、机箱、电源和散热紧密绑定在一起，新硬件要进入数据中心通常需要适配完整服务器形态。这样会使硬件引入慢、成本高，并导致不需要的 CPU、内存或磁盘资源被一起购买和闲置。

论文的基本主张是：数据中心应该把传统服务器拆开，把 CPU、内存、存储和加速器等硬件组织成独立的、网络连接的组件。每个组件都有自己的控制器和网络接口，可以独立运行、独立扩展、独立失效。

## 核心思想

LegoOS 的核心思想不是在现有 Linux 上加一个远端内存或远端存储层，而是提出一种新的 OS 架构：*splitkernel*。

splitkernel 的原则是：硬件资源被拆开后，OS 功能也应该被拆开。传统内核中的进程管理、内存管理、存储管理等功能不再集中运行在一台服务器上，而是拆成多个 monitor。每个 monitor 运行在对应硬件组件上，并本地管理该组件的资源。

LegoOS 将系统分为三类组件：

- `pComponent`：processor component，运行用户进程和 process monitor，管理 CPU cores 和 ExCache。
- `mComponent`：memory component，运行 memory monitor，管理虚拟内存、物理内存、地址映射和内存读写。
- `sComponent`：storage component，运行 storage monitor，管理文件持久化存储。

用户看到的抽象是 `vNode`。一个 vNode 类似虚拟机或虚拟服务器，有自己的 ID、虚拟 IP 和存储挂载点。用户只知道应用运行在某个 vNode 上，不知道内部到底使用了哪些 pComponent、mComponent 或 sComponent。这就是论文所说的 internal execution status is transparent to users：内部执行状态对用户隐藏。

## Splitkernel 与 Coherence

splitkernel 不提供跨组件的 cache coherence。这里的 *coherent* 主要指缓存一致性：当多个处理单元缓存或访问同一份数据时，系统是否自动保证它们看到一致的最新值。

普通多核服务器内部通常有硬件 cache coherence。例如一个 core 写了某个共享内存地址，另一个 core 对应 cache line 会被失效或更新，从而避免继续读到旧值。但在 LegoOS 的目标架构中，不同 pComponent 和 mComponent 通过数据中心网络连接。如果跨网络维护类似 MESI 的全局一致性，需要大量网络流量，并且协议复杂、扩展性差。

因此 LegoOS 选择 non-coherent design：

- 组件内部可以继续使用已有硬件提供的一致性。
- 组件之间不维护透明共享内存一致性。
- 跨组件通信通过显式 message passing 完成。
- LegoOS 不支持跨 processor 的 writable shared memory。

这带来一个重要约束：同一进程内共享内存的线程会被调度到同一个 pComponent；不同进程之间如果需要通信，应使用 socket、RPC 或应用层消息传递，而不是依赖跨 pComponent 的可写共享内存。

## pComponent、Virtual Cache 与 ExCache

LegoOS 最关键的设计挑战是把 processor 和 memory 分离。传统处理器和 OS 假设页表、TLB、内存控制器和主内存都在本地机器内；LegoOS 则把大容量内存和地址映射移动到 mComponent。

为了让 pComponent 不负责虚拟地址到物理地址转换，LegoOS 将 pComponent 上的 L1/L2/L3 cache 组织为 virtual cache，即 virtually-indexed and virtually-tagged cache。cache 用虚拟地址索引和标记，而不是用物理地址。

virtual cache 会带来两个经典问题。

**Synonym problem** 是同一物理地址可能对应多个虚拟地址，导致同一数据在 cache 中出现多个副本。LegoOS 通过不支持跨进程 writable shared memory 避免这个问题。

**Homonym problem** 是不同进程可能使用相同虚拟地址，但这些虚拟地址对应不同数据。例如进程 A 和进程 B 都可能使用 `0x400000`。如果 cache 只看虚拟地址，就可能把 A 的数据错误返回给 B。LegoOS 的解决方法是在 cache line 中加入 ASID，也就是 address space ID。cache 判断命中时使用 `(ASID, virtual address)`，从而区分不同进程的同名虚拟地址。

由于所有应用内存都放到远端 mComponent，单纯依赖网络访问会太慢。LegoOS 因此在 pComponent 上保留少量 DRAM/HBM，作为 `ExCache`。ExCache 可以理解为传统 LLC 后面的一层额外缓存，位置大致是：

```text
L1/L2/L3 virtual cache -> ExCache -> network -> mComponent DRAM/NVM
```

但 ExCache 不是普通 LLC。它有几个特殊点：

- 它使用 pComponent 上的 DRAM/HBM，而不是 CPU 片上 SRAM。
- 它是 virtual cache，用虚拟地址组织。
- hit path 尽量由硬件处理。
- miss path 由 LegoOS process monitor 通过软件处理。
- 它的目的主要是隐藏远端内存网络访问延迟。

pComponent 上这块本地 DRAM/HBM 还会被划分出一部分给 LegoOS 内核自己使用。这里的“给内核使用”不是指内核把 ExCache 当普通应用数据缓存，而是指内核需要本地物理内存存放自己的数据结构，例如线程状态、调度队列、kernel stack、RPC buffer、ExCache tags、FIFO/LRU metadata、vRegion array 缓存副本和 Linux syscall 兼容层状态。LegoOS 启动时保留一段连续物理内存，并固定划分为 ExCache、ExCache metadata 和 kernel physical memory。内核直接用物理地址访问这些本地数据，避免为了访问内核元数据还要去远端 mComponent。

这个设计的一个好处是 pComponent 不需要维护传统地址映射，也不需要访问 TLB。应用访问虚拟地址时，pComponent 可以直接用虚拟地址计算 ExCache set；如果 ExCache hit，就不需要 VA-to-PA translation，也不会遭遇 TLB miss。真正的地址映射和物理内存管理由 mComponent 负责。

## 内存管理与 vRegion

LegoOS 允许一个进程的虚拟地址空间分布在多个 mComponent 上。问题是：pComponent 访问某个虚拟地址时，如何知道应该把请求发给哪个 mComponent？

`vRegion` 用来解决这个问题。LegoOS 把每个进程的虚拟地址空间切成固定大小的大块，例如每块 1 GB。每个已经分配的 active vRegion 都有一个 owner mComponent，负责该地址范围内的内存访问和虚拟内存管理。

可以把一个进程的地址空间理解为：

```text
0 GB - 1 GB   vRegion 0 -> mComponent 1
1 GB - 2 GB   vRegion 1 -> mComponent 2
2 GB - 3 GB   vRegion 2 -> mComponent 2
3 GB - 4 GB   vRegion 3 -> mComponent 5
```

如果程序访问 `1.3 GB` 附近的虚拟地址，pComponent 通过本地缓存的 vRegion array 知道它属于 vRegion 1，于是直接把请求发给 mComponent 2。

论文称这个机制为 two-level distributed virtual memory management。

第一级是粗粒度的 vRegion 管理。每个进程有一个 home mComponent。home mComponent 保存 vRegion array，知道每个 vRegion 归哪个 mComponent 管，也知道每个 vRegion 还剩多少虚拟空间。它负责处理 `brk`、`mmap`、`munmap`、`mremap` 等虚拟内存相关系统调用的高层决策。

第二级是细粒度的 VMA 管理。每个 vRegion owner mComponent 维护自己负责范围内的 vma tree，记录具体哪些虚拟地址范围已经被分配、权限是什么，并管理本地物理内存分配和虚拟到物理地址映射。

这个设计的目的有三个：

- pComponent 可以通过 vRegion array 快速路由内存访问，避免每次访问都问 home mComponent。
- 一个进程的地址空间可以分布在多个 mComponent，提高容量利用率和并行度。
- home mComponent 只做粗粒度管理，owner mComponent 做细粒度管理，避免中心节点成为瓶颈。

## Optimization on Memory Accesses

Section 4.4.2 的优化针对 ExCache cold miss。朴素设计中，每次 ExCache miss 都要访问 mComponent。但很多真实应用会频繁第一次访问新分配的匿名内存，例如 heap、stack 或 anonymous mmap。

匿名内存是没有文件 backing 的进程内存。`malloc`、`brk`、stack growth 和 `mmap(..., MAP_ANONYMOUS, ...)` 通常都属于匿名内存。它的一个重要语义是：新分配时内容应该是 0。OS 不能把别的进程之前用过的数据泄露给当前进程，因此新页面通常表现为 zero-filled page。

LegoOS 利用这个语义优化首次访问：

```text
应用申请匿名内存
-> pComponent 记录该地址范围和权限
-> 应用第一次访问该地址
-> ExCache miss
-> LegoOS 不访问 mComponent
-> 直接在 ExCache 中分配一条全 0 的 cache line
```

论文把这种只存在于 pComponent ExCache、mComponent 还不知道的 cache line 称为 `p-local line`。此时 mComponent 还没有为这块地址分配物理内存，也没有建立地址映射。只有当这条 line 被写脏并从 ExCache 驱逐时，pComponent 才会把 dirty line 发给 owner mComponent；mComponent 在此时分配物理内存、建立映射并保存数据。

这个优化本质上是 delay physical memory allocation until write time。它能减少大量匿名内存 cold miss 的远端网络访问。它之所以安全，是因为 LegoOS 不支持跨 pComponent 的可写共享内存，其他 pComponent 不会同时访问这条只存在于本地 ExCache 的 p-local line。

## Storage Management 与 Stateless Design

LegoOS 支持 POSIX 风格的层次文件接口。用户可以在 vNode 的挂载点下读写目录和文件。但内部 storage monitor 采用简化设计。

`stateless storage server design` 指的是 sComponent 尽量不保存客户端会话状态或进程相关状态。每个 I/O 请求都携带完成操作所需的全部信息，例如 vNode ID、完整路径名、文件 offset、读写长度和操作类型。sComponent 收到请求后，不需要知道某个进程之前 open 了哪个文件，也不需要维护 fd 到文件对象的映射。

传统 Linux 中，`read(fd, ...)` 依赖内核保存的 fd table、open file object、inode、当前 offset 等状态。LegoOS 为了保持 storage monitor 简单，把这些 Linux 风格状态主要放在 pComponent 的 syscall compatibility layer。pComponent 会把 `fd` 翻译成完整路径、offset 和 size，再发给后面的组件。

文件访问路径通常是：

```text
pComponent -> mComponent -> sComponent
```

原因是 LegoOS 把 storage buffer cache 放在 mComponent，而不是 sComponent。sComponent 的控制器内存有限，不适合放大量缓存；mComponent 本来就是内存组件，更适合承担文件缓存。

读文件时，如果 buffer cache 命中：

```text
pComponent -> mComponent -> pComponent
```

如果 buffer cache miss：

```text
pComponent -> mComponent -> sComponent -> mComponent -> pComponent
```

写文件也类似，先写到 mComponent 管理的 buffer cache，fsync 或 flush 时再写到 sComponent。需要注意的是，并不是每个 sComponent 都要绑定一个专属 mComponent。mComponent 和 sComponent 是独立资源池；一个文件的 buffer cache 被放在某个 mComponent 上，而它的持久化数据可以在某个 sComponent 上。论文当前为了简化并避免 coherence traffic，规定一个文件的 buffer cache 只放在一个 mComponent 上。

## Global Resource Management 与 Failure Handling

LegoOS 使用两级资源管理。高层有三个全局资源管理器：

- `GPM`：global process manager，选择新进程运行在哪个 pComponent。
- `GMM`：global memory manager，选择新 vRegion 由哪个 mComponent 管理。
- `GSM`：global storage manager，管理存储资源。

这些全局管理器只做粗粒度决策，并维护近似资源使用信息。细粒度决策由各组件 monitor 本地完成。例如线程调度由 pComponent 本地完成，虚拟/物理内存分配由 mComponent 本地完成。这样可以避免全局管理器进入每次内存访问或每次线程调度的关键路径。

在可靠性方面，论文当前主要处理 mComponent failure。原因是进程内存可能分散在多个 mComponent 上，如果不处理内存组件故障，应用失败概率会增加。LegoOS 没有完整处理 pComponent failure，也把 sComponent 的可靠性更多留给应用或未来工作。

LegoOS 没有简单地做双份完整内存复制，因为这会带来至少 2 倍内存开销。它为每个 vma 使用 primary mComponent、secondary mComponent 和 sComponent 上的 backup log。primary 保存完整内存数据和元数据；secondary 保存小的 append-only log 和 vma tree 副本；后台再把 log 刷到 sComponent。primary 失败时，可以通过 secondary 和 sComponent 上的 log 尝试恢复内存内容。

## 实现

LegoOS 用 C 实现在 x86-64 上，当前原型约 206K SLOC，其中 56K SLOC 是驱动。系统支持 113 个 Linux syscalls、15 个 pseudo-files 和 10 个 vectored syscall opcodes。论文强调支持 Linux ABI 是为了降低采用门槛，让很多数据中心应用可以不修改地运行。

由于没有真实的 disaggregated hardware，论文用普通服务器模拟 pComponent、mComponent 和 sComponent。例如用限制可用 core 的服务器模拟 mComponent 控制器；用限制本地内存的服务器模拟 pComponent，并把本地内存配置为软件管理的 ExCache。

LegoOS 内部通信使用基于 RDMA 的 RPC 框架。pComponent 的 ExCache miss、dirty line writeback、mComponent 请求处理和 sComponent 存储请求都依赖这个低延迟网络栈。论文还实现了 socket-over-RDMA 和传统 TCP/IP socket 栈，供应用使用。

## 实验设置与结果

论文在 10 台服务器上评估 LegoOS。每台机器有两颗 Intel Xeon E5-2620 2.40GHz、128 GB DRAM 和 40 Gbps Mellanox ConnectX-3 InfiniBand，使用 40 Gbps InfiniBand switch 连接。对比 Linux 版本是 v4.9.47。

Figure 8 评估的是 mComponent 的远端内存访问吞吐。实验使用一个很大的空 ExCache，让每次 4 KB memory load 都产生 ExCache cold miss，并访问 mComponent。横轴是 workload threads 数量，纵轴是 KIOPS。图例中的 `1M x 4-worker` 表示 1 个 mComponent、4 个 worker thread；这里的 `M` 是 mComponent，不是 MB。

- mComponent worker 越多，远端内存请求吞吐通常越高。
- 应用线程越多，请求并发度越高，更容易打满 mComponent。
- 使用多个 mComponent 可以分散访问压力，但当总 worker 数达到约 4 后收益开始递减。
- `p-local` 优化显著提高匿名内存首次访问性能，因为它绕过了 mComponent。
- `p-local` 仍然慢于 Linux 本地内存访问，因为它仍需要 trap 到 LegoOS kernel 并设置 ExCache metadata。

Figure 10 使用 PARSEC 中的 BlackScholes、Freqmine 和 StreamCluster，将 LegoOS 和内存充足的单机 Linux 比较。LegoOS 配置为 128 MB ExCache、1 个 mComponent、1 个 worker 和 1 个 sComponent。这个实验主要展示 LegoOS 相对于理想本地 Linux 的 slowdown，不是在扫描 ExCache size。

Figure 11 和 Figure 12 分别评估 TensorFlow 和 Phoenix。横轴写作 `ExCache/Memory Size`，是因为论文把 LegoOS 的 ExCache size 和 Linux-swap baseline 的本地主内存大小设成一样，用于公平比较。对 LegoOS 来说，128/256/512 MB 是 pComponent 本地 ExCache；对 Linux-swap、ramdisk-swap 和 InfiniSwap 来说，这是本地可用 main memory，剩余工作集通过 swap 或远端内存处理。

这个设定并不意味着 ExCache 等于主内存。语义上：

```text
LegoOS:
  ExCache = 本地缓存
  mComponent = 应用真正的大容量内存

Linux-swap:
  local DRAM = 本地主内存
  SSD/ramdisk/remote memory = swap 后备空间
```

这样比较的是：当 processor 旁边只有相同大小的快速本地内存时，不同系统如何处理放不下的工作集。

主要应用结果是：

- 当 ExCache 大约为工作集 25% 时，LegoOS 相比内存充足的单机 Linux 在 TensorFlow 上慢约 1.68 倍，在 Phoenix 上慢约 1.34 倍。
- LegoOS 通常显著好于 SSD swap 和 remote swap。
- LegoOS 的优势来自更轻量的 RDMA RPC、比 Linux paging 更短的路径，以及匿名内存首次访问优化。
- ExCache 管理策略很重要。论文发现 FIFO + free list + piggyback dirty flush and miss fill 的组合最好；LRU 反而因为锁竞争表现较差。
- 使用更多 mComponent 不一定提高应用性能，因为 dirty flush 和 miss fill 不一定发往同一个 mComponent，会削弱 piggyback 优化。
- 内存复制带来约 2% 到 23% 性能开销。

故障分析中，论文基于设备 MTTF 估算等价集群的可靠性。LegoOS 因为组件更简单、资源利用率更高，估算 MTTF 比 monolithic server cluster 提高约 17% 到 49%。但这个分析不包含 transient failures，也不是实物硬件长期运行结果。

## 局限性

这篇论文的限制比较明显：

- 没有真实硬件，只用普通服务器模拟 disaggregated components。
- pComponent 的 virtual cache 和 ExCache 依赖硬件趋势，实际产品化需要专门硬件支持。
- 不支持跨 pComponent 的 writable shared memory，应用模型受到约束。
- 当前主要处理 mComponent failure，没有完整处理 pComponent 和 sComponent failure。
- storage monitor 是简化实现，当前不是论文重点。
- 资源管理和负载均衡仍然初步，不支持内存数据迁移。
- 性能高度依赖应用内存访问局部性、ExCache 命中率、RDMA 网络和 mComponent worker 并行度。

## 阅读过程中的问题与理解

我一开始关注 Section 2.1 中 bad support for heterogeneity 的含义。后来理解为，传统服务器把硬件设备紧密绑定在一个整机形态中，导致 GPU、TPU、FPGA、NVM 等新硬件很难按需独立接入。数据中心常常不得不购买带有整套 CPU、内存、磁盘和主板的新服务器来获得某类加速器，造成资源浪费，也拖慢硬件创新部署速度。LegoOS 希望通过资源解耦，让不同硬件类型独立扩展和独立接入。

另一个关键概念是 coherent。这里主要指跨组件的 cache coherence。普通多核服务器内部由硬件保证多个 core 对共享内存的缓存一致性；LegoOS 不在 pComponent 和 mComponent 之间提供这种全局一致性。它选择显式 message passing，是为了避免在数据中心网络上维护共享内存一致性的巨大流量和复杂度。

我也澄清了 vNode 抽象。用户看到的是类似虚拟服务器的 vNode，而不是一组具体硬件组件。一个 vNode 内部可能使用多个 pComponent、mComponent 和 sComponent；同一个物理组件也可能服务多个 vNode。这种内部执行状态对用户隐藏，降低了使用复杂度。

关于 homonym problem，我的理解是：virtual cache 只看虚拟地址会混淆不同进程的相同虚拟地址。LegoOS 用 ASID 加虚拟地址共同作为 cache line 身份，避免进程 A 的 `0x400000` 被进程 B 的 `0x400000` 错误命中。

关于 pComponent 本地 DRAM/HBM，我一开始容易把 kernel physical memory 和 ExCache 混在一起。更准确的理解是：pComponent 本地内存被分成 ExCache、ExCache metadata 和 kernel physical memory。ExCache 主要缓存应用远端内存数据；kernel physical memory 保存 LegoOS 自己的内核状态；ExCache metadata 则保存 tag、权限位、valid/dirty 状态和替换队列。内核管理 ExCache，但不把 ExCache 当作自己的主要数据存储。

关于 ExCache，我的理解是它很像 LLC 后面的一层 L4 cache，但不能简单等同于 LLC。它是用 pComponent 本地 DRAM/HBM 实现的 virtual, inclusive, software-managed extended cache，主要用于缓存 mComponent 上的远端内存数据。ExCache hit 可以避免网络访问；ExCache miss 则陷入 LegoOS，由 process monitor 从 mComponent 拉取数据或写回 dirty line。

关于 vRegion，我最后把它理解成“进程虚拟地址空间到 mComponent 的粗粒度路由表”。vRegion 解决的是一个进程地址空间分布到多个 mComponent 后，pComponent 如何快速找到目标 mComponent 的问题。home mComponent 保存 vRegion array，owner mComponent 保存该 vRegion 内部的 vma tree 和地址映射。

关于 Optimization on Memory Accesses，我的理解是它利用匿名内存初始为 0 的语义，把匿名内存第一次访问从远端 mComponent 读取变成本地 ExCache 生成 zero line。只有这条 p-local line 被写脏并驱逐时，mComponent 才真正分配物理内存并保存数据。

关于 storage path，我的理解是文件访问通常经过 `pComponent -> mComponent -> sComponent`。mComponent 承担 buffer cache，sComponent 承担持久化存储。cache hit 时不需要访问 sComponent；cache miss 或 fsync 时才访问 sComponent。一个 sComponent 不需要绑定一个专属 mComponent，二者是独立资源池。

最后，Figure 8、Figure 10 和 Figure 11/12 的比较对象也需要区分。Figure 8 测远端内存 miss path 吞吐；Figure 10 测 LegoOS 相比内存充足单机 Linux 的 PARSEC slowdown；Figure 11/12 中 ExCache size 和 Linux main memory size 设成一样，是为了公平比较“processor 旁边快速本地内存容量相同”时不同系统的表现，而不是说 ExCache 在语义上等同于主内存。

## 我的理解

这篇论文的重要性在于，它把资源解耦从硬件设想推进到 OS 架构问题。只把远端内存做成 swap 或 RDMA memory pool，仍然保留了 monolithic server 的基本假设；LegoOS 则直接问：如果 CPU、内存、存储都变成网络连接的独立组件，OS 应该如何组织？

LegoOS 的设计逻辑可以概括为：

- 用 splitkernel 把 OS 功能拆到硬件组件本地，减少远端管理流量。
- 用 vNode 给用户保留类似虚拟服务器的简单抽象。
- 用 non-coherent message passing 避免跨数据中心网络维护共享内存一致性。
- 用 virtual cache 和 ExCache 让 pComponent 在没有本地大容量内存和 TLB 的情况下仍能高效运行应用。
- 用 vRegion 把分布式虚拟地址空间管理变成可路由、可扩展的两级结构。
- 用 mComponent 上的 buffer cache 和 stateless sComponent 简化存储组件。

整体来看，LegoOS 是一篇系统原型论文。它的贡献不在于证明当前硬件上资源解耦已经全面优于传统服务器，而在于提出了一个完整 OS 设计空间：当硬件从 server-centric 变为 component-centric，OS 也应从 monolithic kernel 变为分布在各组件上的 splitkernel。它的很多机制，例如 ExCache、vRegion、p-local line 和 RDMA RPC，都是围绕“减少跨网络访问”和“避免中心化管理”这两个目标展开的。

这篇论文也留下了明确问题：真实硬件如何支持 virtual cache 和远端内存；跨组件故障如何完整处理；没有 writable shared memory 会限制哪些应用；资源管理器如何在大规模集群中做动态负载均衡和内存迁移。这些问题说明 LegoOS 更像是一个开辟方向的原型，而不是可以直接替代 Linux 服务器集群的完整产品系统。

# Jenga: Effective Memory Management for Serving LLM with Heterogeneity

## 基本信息

- 论文：Jenga: Effective Memory Management for Serving LLM with Heterogeneity
- 作者：Chen Zhang, Kuntai Du, Shu Liu, Woosuk Kwon, Xiangxi Mo, Yufeng Wang, Xiaoxuan Liu, Kaichao You, Zhuohan Li, Mingsheng Long, Jidong Zhai, Joseph Gonzalez, Ion Stoica
- 会议：SOSP 2025
- 主题：LLM serving、GPU 显存管理、KV cache、异构 attention、prefix caching

## 文章概述

这篇论文研究的是现代 LLM serving 中的 GPU 显存管理问题。传统 LLM 推理系统通常基于 PagedAttention 管理 KV cache。PagedAttention 将显存切成固定大小的 page，并按请求长度动态分配 page。这个设计在早期 Transformer 模型中很有效，因为这些模型基本是同构的：所有层大多是 full attention，每个 token 在每层产生的 KV cache 大小相同，每一层需要的 page 数量也比较接近。

但论文指出，新一代 LLM 已经不再满足这些假设。现代模型中常见的结构包括 sliding-window attention、local attention、Mamba/state-space layer、VLM 中的 vision embedding 和 cross attention，以及 speculative decoding 中同时运行的小模型和大模型。这些机制导致两类异构性：

- 不同 layer 或 token 类型的缓存大小不同；
- 不同 attention 机制依赖的历史 token 范围不同。

因此，传统固定页、按层均分、统一 LRU 的 KV cache 管理方式会造成显存浪费和 prefix cache 命中率下降。Jenga 的目标就是让 inference engine 的 memory manager 理解这些 layer-specific property，从而更高效地分配、释放和缓存 GPU 显存。

               ## 背景与问题

LLM 推理中，KV cache 是显存占用的主要来源之一。在 decode 阶段，每个新 token 会用自己的 query 去 attend 历史 token 的 key/value，因此系统需要保留历史 token 在各层的 K/V 激活值。为了提高吞吐，serving 系统会 batch 多个请求，但 batch size 越大，需要保存的 KV cache 越多，GPU 显存越容易成为瓶颈。

PagedAttention 的核心假设是同构性。它把每一层分配成固定大小的 page 区域，然后根据请求 token 数量分配 page。这种做法隐含了两个前提：

- 不同层、不同 token 的 embedding/KV cache 大小相同；
- 每一层都以类似方式依赖历史 token，因此需要类似数量的 page。

现代 LLM 打破了这两个前提。例如 Gemma-3 和 Ministral 混合 full attention 与 sliding-window attention；Llama 4 使用 local attention；Jamba/Hymba 混合 attention 和 Mamba；VLM 需要同时管理文本 KV cache 和图像 embedding；speculative decoding 同时维护 draft model 和 target model 的 KV cache。

这种异构性带来两个主要挑战。

第一是显存碎片和浪费。不同缓存对象大小不同，如果用最大 page size 统一管理，小对象会浪费大量空间；如果按类型静态分区，又无法适应运行时请求长度和 token 类型比例的变化。

第二是 prefix caching 变复杂。Prefix caching 的目标是让新请求复用早期请求的公共前缀 KV cache。但在异构模型中，不同层对“前缀命中”的要求不同。Full attention 需要完整前缀，sliding-window attention 只需要窗口内 token，Mamba 可能只需要某个压缩 state。一个 token 只要有任意一层 cache miss，就必须重新经过所有层计算，因此整体 cache hit length 被最差的层限制。

## Figure 2b 的含义

Figure 2b 表达的是：在异构 LLM 中，不同 layer type 的显存占比会随着请求 token 数变化而变化。

可以用下面的简化公式理解：

```text
M_full(n)   = full-attention 层数 * 每 token KV 大小 * n
M_window(n) = sliding-window 层数 * 每 token KV 大小 * min(n, window_size)
M_mamba(n)  = mamba 层数 * 每序列 state 大小
```

对于短请求，Mamba state 虽然数量少，但单个 state 可能很大，因此 Mamba 层的显存占比可能较高。随着请求变长，full attention 的 KV cache 按 token 数线性增长，最终会成为主要显存消费者。Sliding-window attention 一开始也随 token 数增长，但超过 window size 后不再继续按全文增长。

这说明异构模型不能用固定比例给各层分配显存。短请求和长请求的瓶颈层可能完全不同，静态分区会造成一部分层显存不足、另一部分层显存空闲。

## Jenga 的核心思想

Jenga 的核心是提出一个 layer-property-aware memory manager。它不再把所有层当成同一种 attention，而是为每种 layer type 定义统一接口：

```python
class LayerProperty:
    def page_size()
    def active_pages(request, length)
    def possible_prefix(is_hit)
```

这三个接口分别表示：

- `page_size()`：该层每个 token 或 state 需要多少显存；
- `active_pages()`：当前请求在后续生成中真正还会用到哪些 page；
- `possible_prefix()`：在给定 cache 状态下，哪些 prefix length 可以算作有效命中。

基于这个接口，Jenga 设计了三部分机制：

- LCM allocator：支持不同大小 page 的动态共享，减少碎片；
- layer-specific allocator：只给 active pages 分配显存，并及时释放 inactive pages；
- layer-specific prefix cache manager：为不同 attention 机制定制 cache hit 和 eviction 策略。

## LCM Allocator

Jenga 使用两级分配器解决不同 page size 之间的碎片问题。全局 allocator 先把 GPU KV-cache memory 切成大页，大页大小是所有 layer page size 的最小公倍数。每种 layer type 再从全局 allocator 申请大页，并把大页切成自己需要的小页。

例如 image token page size 是 256，text token page size 是 384，那么 Jenga 使用：

```text
LCM(256, 384) = 768
```

作为大页大小。一个 768 的大页可以切成 3 个 image 小页，也可以切成 2 个 text 小页。

这个设计的好处是：

- 大页在全局池中统一管理，可以在不同 layer type 之间动态流动；
- 小页由各 layer-specific allocator 管理，可以匹配对应 layer 的实际 page size；
- 当一个大页内的小页全部释放后，大页可以归还给全局 allocator；
- attention kernel 仍然可以看到普通的 page id 和 page size，因此 Jenga 可以尽量复用 PagedAttention kernel。

## 为什么原来的 Memory Layout 不适合 Jenga

论文中说：

> However, this memory layout is not suitable for Jenga, because different layer types can have different number of layers.

这句话的意思是：PagedAttention 的原始布局是先按 layer 切分显存，再在每个 layer 内切 page。这对同构 Transformer 合适，因为每层 KV cache 大小相同、page 数量相似。

但 Jenga 面对的是异构模型。例如一个模型可能有：

```text
10 个 full-attention layers
52 个 sliding-window layers
```

长请求时 full attention 的 KV cache 持续增长，sliding-window attention 超过窗口后不再增长。如果显存已经固定分给每一层，那么 sliding-window 层空出来的显存很难直接给 full-attention 层使用。结果是一边缺显存，一边有空闲。

所以 Jenga 改成 page-layer partition：先由全局 page allocator 管理大页，再根据 layer type 映射出各层需要的视图。这样显存可以在 full attention、sliding-window attention、Mamba、vision cache 等不同类型之间动态交换。

## Prefix Caching 的复杂性

Prefix caching 是指：如果一个新请求和旧请求共享前缀，就复用旧请求保留下来的 KV cache，避免重新 prefill 公共前缀。

异构 attention 让 prefix caching 变复杂，主要有两个原因。

第一，整体 cache hit length 被最短命中的层限制。假设 full-attention 层只能命中到 3 个 token，而 Mamba 层能命中到 5 个 token，那么整个模型只能复用前 3 个 token。因为第 4 个 token 在 full-attention 层缺 cache，重算这个 token 时所有层都要重新执行。

第二，不同 layer type 为了达到同样的 prefix hit length，需要缓存的内容不同。Full attention 需要完整 prefix 的所有 token；sliding-window attention 只需要窗口内 token；Mamba 可能只需要一个压缩 state 或若干 checkpoint state。因此，如何在不同层之间分配 cache memory，不是固定答案，而是取决于请求长度和 workload 的 prefix 复用模式。

这里还有 batch size 和 prefix cache size 的权衡。GPU 显存既可以用来让当前 running requests 跑更大的 batch，也可以用来保存已经完成请求的 prefix cache。缓存越多，未来 cache hit 可能越多；但当前可用于 batch 的显存越少，batch size 可能下降。

## Cached Page Pool 与 Common Page Pool

Jenga 的普通 cached page pool 是 opportunistic cache。论文说 Jenga 使用所有没有分配给 running requests 的显存维护 cached page pool，意思是：

```text
当前请求优先使用显存；
剩余显存用于 prefix cache；
如果当前请求需要更多显存，就从普通 cached page pool 中 eviction。
```

因此，普通 cached page pool 没有预先做固定比例的 batch/cache partition。

但 Jenga 后面又引入了 common page pool。Common page pool 用来保存未来很可能复用的公共前缀 page。这部分 page 更受保护，不会轻易为了扩大 batch size 被淘汰。

所以更准确地说：

- 普通 cached page pool：batch 优先，剩余显存用于 cache；
- common page pool：保护高价值 prefix page，在 cache hit 和 batch size 之间做显式权衡。

Jenga 会模拟不同 common page pool size 下的总执行时间，并选择估计最优的 pool size。

## Figure 11 的含义

Figure 11 解释的是 Jenga 如何为不同 attention layer 定制 LRU eviction。它关注的不是 cache hit 本身，而是 last access time 应该如何更新。

图中有两个请求：

```text
Request 1: 输入 A B C D，输出 E F
Request 2: 输入 A B C D G，输出 H
```

假设 sliding window size = 2。

对 full attention 来说，生成新 token 时需要访问所有历史 token。因此每次请求访问公共前缀时，`A B C D` 都是真正有用的历史 KV cache，它们的 last access time 应该更新。

但对 sliding-window attention 来说，生成下一个 token 只需要最近两个 token。即使 `A B` 属于请求前缀，只要它们已经在窗口外，后续生成就不会再访问它们。因此 Jenga 不应该把 `A B` 视为“最近使用”。它们的 last access time 不更新，就会在 LRU 中变老，更容易被淘汰。

Figure 11 的核心是：**不同 attention 机制下，“最近使用”的含义不同**。Jenga 只更新 `active_pages()` 返回的 page 的 last access time，从而让无用 page 更早被淘汰，有用 page 更容易留下。

## Section 6.3：不同 Layer 的定制方式

Section 6.3 是在说明 Jenga 如何把统一的 `LayerProperty` 接口套到不同 layer type 上。

对于 full attention：

- `active_pages` 是整个 prefix 的所有 page；
- `possible_prefix` 要求从开头连续缓存，缺中间任何 token 都不能算完整命中。

对于 sliding-window attention：

- `active_pages` 是最近 window size 个 token 的 page；
- `possible_prefix` 不要求完整 prefix 全部缓存，只要该 prefix 对应窗口内的 page 存在即可。

对于 Mamba：

- Mamba state 通常比单 token KV cache 大；
- 如果每个 token 都缓存 state，会占用大量显存；
- Jenga 只缓存周期性状态，例如每 512 个 token 缓存一次；
- 因此只有 512 倍数长度附近的 prefix 能直接命中。

对于 local attention：

- 序列被划分成固定 chunk；
- 每个 token 只 attend 当前 chunk 内 token；
- `active_pages` 是当前 chunk 内相关 page；
- `possible_prefix` 只要求当前 chunk 依赖的 page 命中。

对于 vision embedding cache 和 vision cross-attention cache：

- 一张图通常对应多个 image tokens；
- 如果只淘汰其中一个 image token，可能导致整张图的 vision encoder 结果都无法完整复用；
- 因此 Jenga 倾向于按 image 粒度设置相同 eviction priority，让一张图的 page 一起保留或一起淘汰。

这一节的意义是证明：Jenga 的抽象不是只适用于某一个模型，而是可以覆盖 full attention、sliding window、Mamba、local attention 和 VLM 等多种现代 LLM 结构。

## 实验结果

Jenga 在 vLLM 上实现，约 4000 行 Python 代码。实验模型包括 Llama 3.2 Vision、Gemma-3、Ministral、Llama 4、Jamba、Character.ai 风格模型、PyramidKV，以及 speculative decoding 场景。

主要结果如下：

- 在 H100 上，Jenga 相比 vLLM 吞吐最高提升 1.73 倍，平均提升 1.46 倍；
- 在 L4 上，Jenga 相比 vLLM 吞吐最高提升 2.16 倍，平均提升 1.65 倍；
- 对普通同构 Llama 3.1，Jenga 几乎没有额外开销；
- 对 Llama 4，Jenga 通过释放 inactive pages 显著提高最大上下文长度，在 8 张 H200 上从 vLLM 的 3.7M token 提升到 14.7M token；
- 对 VLM，Jenga 通过缓存 vision embedding 避免每个 chunked prefill step 都重跑 vision encoder，从而显著提升吞吐；
- 对 speculative decoding，Jenga 可以自动处理 draft model 和 target model 的不同 KV cache size。

## 局限性

Jenga 的一个限制是，它依赖正确的 layer property 定义。对于已经支持的 attention 类型，系统可以自动解析模型结构；但如果未来出现新的 attention 或 state-space 机制，仍然需要补充对应的 `active_pages` 和 `possible_prefix` 规则。

另一个限制是 common page pool 依赖 workload 的 prefix 复用规律。如果请求之间几乎没有公共前缀，或者公共前缀不在短时间内重复出现，那么 common page pool 的收益会下降。

此外，Jenga 主要解决单个 inference engine 内部的 GPU memory management。它没有完整处理跨节点 routing、分布式 KV cache placement、CPU/SSD offloading 等更大规模 serving 问题。

## 阅读过程中的问题与理解

我在阅读中首先困惑的是 Figure 2b。后来理解为，这张图不是在展示绝对显存大小，而是在展示不同 layer type 的相对显存占比如何随请求长度变化。短请求时 Mamba 可能占比高，长请求时 full attention 会逐渐占主导，sliding window 则在超过窗口后趋于稳定。

我也澄清了 prefix caching 中 cache hit dependency 的含义。模型的有效 cache hit length 被命中最短的 layer 限制，因为任意一层缺失都会导致该 token 需要重新经过所有层计算。因此不能只让某类层缓存得很好，而另一类层缓存很差；否则整体复用长度仍然很短。

关于 batch size 和 prefix cache size 的关系，我的理解是：Jenga 的普通 cached page pool 没有固定 balance，它优先满足 running requests，剩余显存才用于 cache。但 common page pool 是一个专门保护高复用 prefix page 的机制，它确实在 cache hit 和 batch size 之间做了有限的显式权衡。

关于 memory layout，我理解到 PagedAttention 的 layer-first layout 会把显存锁在各层区域中，而 Jenga 需要不同 layer type 之间动态交换显存。因此 Jenga 必须使用更灵活的 page-layer layout 和 LCM allocator。

Figure 11 帮助我理解了 customized eviction。对 full attention，整个 prefix 都是 active；对 sliding-window attention，窗口外 token 虽然属于请求前缀，但对未来生成没有用，所以不应刷新 LRU 时间。Jenga 的关键就是让 LRU 的“最近使用”符合每种 layer 的实际访问模式。

最后，Section 6.3 可以理解为把同一个接口应用到不同层的例子。Full attention、sliding window、Mamba、local attention 和 vision cache 都有不同的 active page 和 possible prefix 定义，但可以被同一套框架管理。

## 我的理解

这篇论文的核心贡献不是提出新的模型结构或 attention 算法，而是指出现代 LLM serving 的 memory manager 必须理解模型结构。PagedAttention 的固定页和同构层假设在早期 Transformer 中成立，但在混合 attention、VLM、Mamba、speculative decoding 等场景中会造成显存浪费。

Jenga 的设计逻辑比较清晰：

- cache size 异构，所以需要 LCM allocator 减少碎片；
- attention dependency 异构，所以需要 `active_pages()` 判断哪些 page 真正有用；
- prefix hit rule 异构，所以需要 `possible_prefix()` 定义每类层的命中条件；
- running batch 和 prefix cache 竞争显存，所以需要 cached page pool 和 common page pool 区分普通缓存与高价值缓存。

整体来看，Jenga 是一篇典型系统论文。它把一个模型架构趋势，也就是 LLM 越来越异构，转化为 inference engine 的内存管理问题。它的意义在于为现代 LLM serving 提供了一个更通用的 KV cache 管理框架，使系统能在不修改 attention kernel 太多的情况下，提高显存利用率、batch size 和吞吐。

# Rearchitecting the Thread Model of In-Memory Key-Value Stores with μTPS

## 基本信息

- 论文：Rearchitecting the Thread Model of In-Memory Key-Value Stores with μTPS
- 作者：Youmin Chen, Jiwu Shu, Yanyan Shen, Linpeng Huang, Hong Mei
- 会议：SOSP 2025
- 主题：in-memory key-value store、thread architecture、RDMA、cache efficiency、contention mitigation、auto-tuning

## 背景与问题

现代 in-memory key-value store（KVS）通常运行在用户态，并通过 RDMA、DPDK、libibverbs 等 kernel-bypass 网络栈处理请求。服务端 worker thread 固定绑定到 CPU core 上，通过 busy polling 接收客户端的 `GET`、`PUT`、`DELETE` 等请求，然后访问内存中的 index 和 value storage。由于网络和内存设备已经非常快，KVS 的单次操作预算可能只有几百纳秒到几微秒。

现有高速 KVS 大多使用 run-to-completion 的 non-preemptive thread-per-queue（NP-TPQ）模型。每个 worker thread 从网络队列取请求后，会从头到尾完成整个请求：

```text
poll network buffer -> parse request -> index lookup -> copy/update value -> send response
```

这种模型减少了 OS 调度和上下文切换开销，但它也把访问模式完全不同的子任务放在同一个线程中执行。网络 buffer 和 hot items 往往可以驻留在 LLC 中，而 index traversal 和 data access 会访问更大的内存空间，产生大量 cache miss。把它们混在同一个 run-to-completion 函数里，容易造成 cache pollution 和 cache thrashing。

论文的核心问题是：**在几十 Mops/s 的高速 KVS 中，传统 run-to-completion 线程模型是否仍然合适？**

## 核心思想

μTPS 重新引入 thread-per-stage（TPS）思想，但它不是传统意义上按功能拆成很多 stage，而是按 cache residency 拆成两层：

- **Cache-resident layer（CR layer）**：处理网络 buffer、请求解析、响应发送，以及部分 hot items。目标是让这些频繁访问的数据尽量留在 CPU cache 中。
- **Memory-resident layer（MR layer）**：处理完整 index 和完整 data items。目标是通过 batching、prefetching、coroutine 等机制摊薄 cache miss 成本。

因此，μTPS 的重点不是像传统 TPS 那样用 DRAM buffer 隐藏慢磁盘或慢网络 I/O，而是把 cache-friendly 的工作和 cache-unfriendly 的工作拆开，减少缓存相互污染和写热点竞争。

## 为什么 CAT 下 DDIO 提升有限

论文先分析了一个自然但不充分的思路：既然 Intel DDIO 会把 NIC 收到的数据放进 LLC 的若干 reserved ways，那么能否用 Cache Allocation Technology（CAT）隔离这些 ways，从而避免网络 buffer 被普通 CPU 代码污染？

实验显示，NP-TPQ-CAT 的提升很小。原因是 DDIO reserved ways 并不是严格的网络 buffer 隔离区。DDIO 在 cache miss 分配时通常使用固定 ways，但如果某个 cache line 已经被 CPU load 到其他 LLC ways 中，DDIO 可以直接访问或修改该 cache line。

在 run-to-completion 模型中，worker thread 会主动读取网络 buffer 来 poll 和 parse 请求。这会把网络 buffer cache line 带入 worker 可用的普通 LLC ways。随后同一个 worker 又执行 index traversal 和 data access，这些随机内存访问会把刚才的网络 buffer cache line 挤出 cache。也就是说，CAT 只能部分隔离 cache space，不能解决同一个执行流把网络处理和内存访问混跑的问题。

μTPS 的解决方式是隔离执行流：CR threads 主要处理网络 buffer 和 hot items，MR threads 处理完整 index 和 data。这样 MR layer 的 cache miss 不会持续冲刷 CR layer 的缓存工作集。

## μTPS 的主要设计

### Reconfigurable RPC

传统 RDMA/RPC 系统常常把客户端连接和服务端 worker thread 绑定在一起。worker 数量变化时，客户端也要知道新的路由关系，重配置成本很高。μTPS 需要根据负载动态调整 CR layer 的线程数，因此提出 reconfigurable RPC。

其核心是服务端使用一个共享 receive buffer / Shared Receive Queue（SRQ）。客户端只需要把请求发送到服务端共享队列，不需要知道服务端当前有多少 worker。服务端 worker 根据 receive buffer slot 编号分工：

```text
slot m -> worker (m mod n)
```

其中 `n` 是当前 CR layer worker 数量。若 auto-tuner 把 `n` 从 4 改成 2，客户端无需同步任何信息，服务端只需要更新全局变量并让 worker 在安全点切换即可。

论文还使用 Multi-Packet Receive Queue（MP-RQ）降低 post receive 开销。普通 receive queue 中，一个 receive work request 通常对应一个小请求；MP-RQ 则让一个较大的 receive buffer 容纳多个 packet/request，减少管理线程频繁 post receive 的 CPU 和队列管理开销。

### Resizable Cache

CR layer 会缓存 hot items。论文没有使用传统 LRU，因为 LRU 在高速 KVS 中会带来很重的 bookkeeping 开销。μTPS 采用 hot-set-based 方法：后台线程采样最近访问 key，使用 count-min sketch 和 min heap 找出 hot set，然后通过 epoch-based 方法原子切换 cache 内容。

这个 cache 是 resizable 的。cache 太小会错过热点，cache 太大又会占用过多 LLC，并增加 miss path 的负担。因此 auto-tuner 会根据吞吐反馈调整 hot set 中真正放入 CR layer 的 item 数量。

### Memory-Resident Layer 与 Batched Indexing

MR layer 管理完整 index 和完整 data items。它处理 CR layer 转发来的 cold 或 miss 请求。由于这一层会进行大量随机内存访问，论文使用 batched indexing 来隐藏 cache miss。

Batched indexing 不是简单地批量调用 `lookup`。它的关键是把多个 index lookup 的 pointer chasing 交错执行：

```text
request A: prefetch next node, yield
request B: prefetch next node, yield
request C: prefetch next node, yield
return to A: prefetched cache line may have arrived
```

μTPS 用 C++ coroutine 表示每个 indexing operation。MR worker 从 CR-MR queue 一次取一批请求，为每个请求创建或推进一个 coroutine。在一个 coroutine 发起 prefetch 后，它主动让出 CPU，worker 切换到另一个 coroutine。这样可以用其他请求的计算时间隐藏 DRAM 访问延迟。

### CR-MR Queue

CR-MR queue 负责 CR layer 和 MR layer 之间通信。它是一个 multi-producer multi-consumer 设计：每对 CR thread 和 MR thread 之间有一个专用 lock-free ring buffer。CR threads 以 round-robin 方式把 miss 请求推给 MR threads，以平衡 MR layer 负载。

每个 queue slot 可以容纳多个请求，从而把跨核通信、队列 push/pop 和 completion 通知的固定成本摊到一批请求上。Figure 12 中的 ablation 显示，batching 将 μTPS-T 和 μTPS-H 的吞吐分别提高了 51.6% 和 93.7%。

### Auto-tuner

由于 μTPS 的线程都 pinned 到 cores 上，不能依赖 OS scheduler 来动态平衡每个 stage 的资源。μTPS 在存储系统内部实现 auto-tuner，负责调整：

- CR layer 和 MR layer 的 worker thread 数量；
- LLC ways 在两层之间的分配；
- CR layer hot cache 的大小。

auto-tuner 通过吞吐反馈进行分层搜索。它先在给定 cache size 下搜索最佳线程分配，再搜索不同 cache size；LLC allocation 则独立调整。论文中动态 workload 变化后，auto-tuner 约 0.9 秒完成重配置，并且请求处理不会停机。

## 实现与评估

论文实现了两个 KVS：

- **μTPS-H**：使用 libcuckoo 作为 hash index，主要支持 point query。
- **μTPS-T**：使用 MassTree 作为 tree index，支持 point query 和 range query。

实验平台包含 200 Gbps RDMA 网络，服务端使用 Intel Xeon Gold 6330，默认在单 NUMA node 的 28 个 cores 上测试。对比系统包括 BaseKV、eRPC-KV、RaceHash 和 Sherman。其中 BaseKV 与 μTPS 使用相同优化，但采用 run-to-completion 线程模型。

主要结果如下：

- 相比 run-to-completion 的 BaseKV，μTPS 整体获得 1.03 到 5.46 倍 speedup。
- 在 skewed workload、read-intensive workload、tree index 和较大 item size 下，μTPS 的优势更明显。
- 对 MassTree/tree index，μTPS-T 的收益通常高于 hash index，因为 tree traversal 有更多 pointer chasing 和 cache miss。
- 在 uniform workload 或 hash index、小 item 场景下，μTPS 的收益较小；部分场景中 eRPC-KV 可接近甚至超过 μTPS。
- μTPS 的额外延迟较小。跨核通信约为 100ns，并通过 batching 摊薄。
- Figure 12 表明 batching 对性能很重要：batch size 从 1 增大到较大值时，μTPS-T 从约 30 Mops/s 提升到约 45 Mops/s，μTPS-H 从约 37 Mops/s 提升到约 72 Mops/s。

## 局限性

μTPS 的收益依赖 workload 和硬件环境。它最适合高速网络、用户态轮询、请求速率极高、cache miss 或写竞争明显的 KVS。对于 uniform workload、hash index、小 item 和轻量 lookup 场景，run-to-completion 或 eRPC 风格设计已经很强，μTPS 的额外层间通信可能抵消部分收益。

此外，μTPS 依赖 RDMA、DDIO、CAT、core pinning 等硬件和系统特性。它的设计复杂度明显高于普通 KVS，包括 reconfigurable RPC、CR-MR queue、resizable cache 和 auto-tuner。自动调参虽然不阻塞请求处理，但 0.9 秒级别的收敛时间也意味着它更适合负载变化不太频繁的生产环境。

## 阅读过程中的问题与理解

我首先澄清了 KVS 中 thread 的含义。这里的 thread 指服务端 KVS 进程中的 worker thread，而不是客户端本身。客户端发起 `GET` 或 `PUT` 请求，服务端 worker thread 负责从网络队列取请求、访问内存 index 和 value storage，并返回响应。

我也确认了论文中的 KVS 运行在用户态，而不是内核态。初始化阶段仍然需要内核和驱动配置 RDMA queue、注册 memory region、pin memory 等，但请求处理 fast path 通过 kernel-bypass 网络栈在用户态完成，不经过传统内核 socket 路径。

关于 traditional TPS 和 μTPS 的区别，我的理解是：传统 TPS 用 DRAM queue/buffer 隐藏慢 I/O，例如磁盘和早期网络；μTPS 面对的是现代 RNIC 和内存 KVS，I/O 已经很快，瓶颈转移到 cache miss、contention 和跨核通信。因此 μTPS 复用 TPS 的“分阶段”思想，但目的不是隐藏慢 I/O，而是隔离不同 cache 行为的执行流。

关于 OS scheduler 的句子，我的理解是：传统 TPS 中的 thread pool 通常由 OS scheduler 负责运行和抢占，适合会 block/sleep 的慢 I/O 系统。μTPS 的 worker threads pinned 到 cores 并 busy poll，不适合依赖 OS scheduler。它必须由 KVS 自己在用户态决定某些 core 属于 CR layer 还是 MR layer。

关于 reconfigurable RPC，我的理解是：它的关键不是让 RPC 语义可变，而是让服务端接收路径能在 worker 数量变化时不用通知客户端。共享 receive buffer 加 slot modulo 分工，使线程数变化只需要服务端更新 `n`。

关于 batched indexing，我的理解是：它用 coroutine 把多个 index lookup 交错执行，利用 prefetch 和 yield 隐藏 pointer chasing 的 DRAM 延迟。这与普通 batching 的区别在于，它不是只减少函数调用或队列开销，而是直接针对 index traversal 的随机访存延迟。

## 我的理解

这篇论文的价值在于，它把高速 KVS 的瓶颈从“网络和 I/O”重新定位到“cache 行为和线程模型”。在现代 RDMA/KVS 场景中，run-to-completion 虽然减少 OS 开销，但把 network buffer、hot items、full index 和 data access 混在一起，导致 cache-friendly 和 cache-unfriendly 工作互相干扰。

μTPS 的设计逻辑比较清楚：

- network buffer 和 hot items 应该由 CR layer 管理，尽量保持 cache residency；
- full index 和 data items 应该由 MR layer 管理，并用 batching/prefetch/coroutine 隐藏 memory latency；
- CR-MR queue 和 batching 用来控制 stage separation 带来的跨核通信开销；
- reconfigurable RPC 让 CR layer 线程数可以动态变化，而无需客户端参与；
- auto-tuner 在用户态重新分配 pinned cores，以适应 workload 变化。

整体来看，μTPS 是一篇典型系统架构论文。它并不只是优化某个 KVS 数据结构，而是挑战高速 KVS 中“一个 worker 从头到尾处理一个请求”的默认线程模型。它的启发是：当设备足够快时，线程模型本身会决定 cache locality、contention 和 schedulability。
