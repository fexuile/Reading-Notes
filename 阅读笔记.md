# PrefillOnly: An Inference Engine for Prefill-only Workloads in Large Language Model Applications

## 基本信息

- 论文：PrefillOnly: An Inference Engine for Prefill-only Workloads in Large Language Model Applications
- 作者：Kuntai Du, Bowen Wang, Chen Zhang, Yiming Cheng, Qing Lan, Hejian Sang, Yihua Cheng, Jiayi Yao, Xiaoxuan Liu, Yifan Qiao, Ion Stoica, Junchen Jiang
- 会议：SOSP 2025
- 主题：LLM inference engine、prefill-only workload、KV cache 管理、调度、prefix caching

## 背景与问题

传统 LLM 推理系统主要面向生成式任务，例如聊天、代码生成和摘要。这类任务通常包含两个阶段：先通过 prefill 处理完整输入 prompt，再通过多轮 decode 逐 token 生成输出。因此，现有推理引擎通常会为所有层、所有历史 token 保存 KV cache，以便后续 decode 复用。

本文观察到，LLM 正越来越多地用于推荐、信用审核、数据标注等判别式任务。在这些任务中，系统通常不需要模型生成一段长文本，而只需要模型对一个候选 token 或一组候选 token 给出概率分布。例如推荐系统只需要 `P(Yes)` 和 `P(No)`，并将 `P(Yes)` 作为推荐分数。论文将这类请求称为 *prefill-only request*，将对应负载称为 *prefill-only workload*。

这类负载有两个重要特点：

- 每个请求只需要生成一个 token，因此不需要多轮 decode。
- 输入通常很长，例如用户画像、浏览历史、信用记录或长文档。

现有 LLM 推理引擎没有充分利用这两个特点。它们仍然假设请求可能继续 decode，因此保存完整 KV cache，导致长输入时显存压力很大；同时，它们通常使用不感知作业完成时间的调度策略，无法利用 prefill-only 请求完成时间更可预测这一性质。

## 核心思想

PrefillOnly 的核心观点是：如果请求只需要 prefill，那么为未来 decode 保存完整 KV cache 是不必要的。系统可以重新设计显存管理和调度策略，以支持更长输入并提升吞吐。

论文提出的系统 PrefillOnly 主要包含三项技术：

1. **Hybrid Prefilling**
2. **Suffix KV Cache Discarding / Offloading**
3. **Shortest Prefill First with Continuous Prefill Time Estimation**

## Hybrid Prefilling

直觉上，prefill-only 请求不需要保存所有层的 KV cache。当前层 attention 计算完后，该层 K/V 对当前请求通常就不再有用，因为之后不会有 decode token 回来访问这些历史 K/V。

但论文指出，简单地“只保留一层 KV cache”并不能显著提升最大输入长度。原因是 LLM forward 过程中还有大量临时中间张量，尤其是 MLP/linear 层中的 intermediate tensors。这些张量可能比单层 KV cache 更大，仍然会造成显存峰值。

为此，PrefillOnly 使用 hybrid prefilling：

- attention 层不切块，保持完整上下文计算，避免降低 attention kernel 性能；
- 非 attention 层，特别是 linear/MLP 层，按 chunk 处理，从而降低中间张量峰值显存；
- 通过 `torch.compile` 修改计算图，将连续 linear 操作组合成 virtual layer，再对该 virtual layer 分块执行；
- 进一步使用输出预分配和 in-place 计算减少额外显存开销。

这个设计的关键是，它减少了 MLP 中间张量的显存峰值，同时不破坏 attention 层对完整上下文的处理。

## Suffix KV Cache Discarding / Offloading

PrefillOnly 还需要兼顾 prefix caching。许多 prefill-only 场景中，不同请求可能共享很长的前缀，例如同一用户画像加不同候选帖子。保留前缀 KV cache 可以让后续请求复用已有计算，减少延迟和计算量。

因此，PrefillOnly 的策略不是丢弃所有 KV cache，而是：

- 尽量保留 prefix token 的 KV cache，用于未来 prefix caching；
- 对超出 GPU 缓存容量的 suffix token KV cache 进行丢弃或 offload；
- 当前实现主要是 discard，论文指出也可以结合 LMCache 或 Mooncake Store 等系统进行 CPU/外部存储 offload。

Hybrid prefilling 是 suffix KV cache discarding/offloading 的基础。因为 attention 层保持不切块，prefill 可以在一次完整 forward 中完成；系统可以在每层计算过程中丢弃或卸载不需要长期保留的 suffix KV cache，而不需要像 chunked prefilling 那样反复 reload 或 regenerate 前面 chunk 的 K/V。

## 调度：Shortest Prefill First with Continuous Estimation

Prefill-only 请求的输出长度固定为 1，因此比普通生成式请求更容易估计完成时间。论文利用这一点，采用类似 shortest job first 的策略，即优先处理预计 prefill 时间最短的请求。

但 prefix caching 会让请求的实际 prefill 时间动态变化。如果某个请求的前缀已经在 cache 中，那么它只需要计算未命中的 suffix，执行时间会显著缩短。反之，如果 cache 被驱逐，请求时间又会变长。

因此，PrefillOnly 提出 continuous prefill time estimation：

- 每次调度前，重新估计等待队列中所有请求的 prefill 时间；
- 估计时考虑输入长度和 prefix cache 命中 token 数；
- 使用 cache-miss token 数量作为 prefill 时间的近似指标；
- 加入 `-lambda * queue_time` 项，避免短请求持续插队造成长请求饥饿。

这种调度同时实现了两个目标：降低平均延迟，并提高 prefix cache 命中率。

## 实验设置与结果

论文在 vLLM 基础上实现 PrefillOnly，核心代码约 4.6k 行 Python。实验覆盖：

- 4 种硬件设置：NVIDIA L4、A100、H100 PCIe、H100 NVLink；
- 3 个 LLM 模型；
- 2 类模拟负载：post recommendation 和 credit verification；
- baselines 包括 PagedAttention、chunked prefilling、pipeline parallelism 和 tensor parallelism。

主要结果如下：

- 在不增加平均延迟和 P99 延迟的情况下，PrefillOnly 可以支持约 1.4 到 4.0 倍更高 QPS。
- 相比非并行 baseline，PrefillOnly 可将最大输入长度提高最多约 5 倍。
- Hybrid prefilling 显著提升 maximum input length，同时避免 chunked prefilling 对 attention kernel 和吞吐的损害。
- 在高 QPS 下，PrefillOnly 的吞吐和延迟优势更明显；在低 QPS 下，并行方案有时有更低单请求延迟，但由于通信和同步开销，扩展到高负载时吞吐较差。

实验中的 routing 也值得注意。对于 PrefillOnly 和非并行 baseline，系统在每张 GPU 上启动一个完整 inference engine 实例，并使用 user-id-based routing：同一用户的请求固定发送到同一实例，以提高 prefix cache 复用率；不同用户按 round-robin 分配以平衡负载。

## 局限性

论文也讨论了几个限制：

- 当前实现主要丢弃 suffix KV cache，而不是完整实现 offloading，因此可能损失未来请求复用 suffix cache 的机会。
- 当 workload 具有大量长前缀共享，并且 prefix cache 空间本身成为瓶颈时，tensor parallel 或 pipeline parallel 这类并行化方案可能更合适，因为它们可以把 KV cache 分摊到多张 GPU 上。
- PrefillOnly 的调度主要在单个 inference engine 内部完成。大规模部署中，可能需要在 router 层做全局 shortest prefill first routing。
- 论文主要优化吞吐，对极致单请求延迟的优化还有空间。

## 阅读过程中的问题与理解

阅读这篇论文时，我一开始的主要困惑集中在 LLM inference 的基本 workflow 上。后来我把普通生成式推理理解为两个阶段：**prefill** 是一次性处理完整 prompt，建立每一层 attention 的 K/V，并得到第一个输出 token 的概率分布；**decode** 是在此之后逐 token 生成，每一步只输入新生成的 token，同时复用历史 KV cache。这个区分解释了为什么 prefill-only workload 特殊：它只需要 prefill 结束时的一个 token 概率分布，不需要后面的 decode 循环。

我也进一步澄清了 KV cache 的含义。KV cache 存储的不是训练阶段的东西，也不是模型参数，而是当前推理请求中，由训练好的 `W_K` 和 `W_V` 根据输入 hidden states 计算出来的 K/V 激活值。训练学到的是 `W_Q`、`W_K`、`W_V` 这些投影矩阵；推理时每个 token、每一层都会临时算出自己的 Q/K/V。KV cache 保存的是 K 和 V，因为未来 decode token 会用自己的 Q 去 attend 历史 K/V。

另一个关键问题是：为什么会有很多层 attention，以及 PrefillOnly 所说的“一层 KV cache”中的“层”指什么。这里的层指 Transformer layer 或 Transformer block 中的 attention 层。普通生成式推理必须保存所有层的 KV cache，因为新 token 在 decode 时会依次经过所有 Transformer layers，并且在第 `l` 层需要访问历史 token 在同一第 `l` 层的 K/V。PrefillOnly 只做 prefill，因此某一层 attention 算完后，该层 K/V 对当前请求通常不再需要，所以理论上只需要保留当前正在计算的一层 K/V。

我还把 prefix caching 和 suffix KV cache discarding/offloading 的关系理清了。Prefix caching 是指多个请求共享同一个 prompt 前缀时，系统保留这个前缀的 KV cache，让后续请求复用。例如同一个用户画像加不同候选帖子时，用户画像就是共享 prefix。PrefillOnly 不能简单丢弃全部 KV cache，因为这样会损失 prefix caching 的收益；它选择尽量保留 prefix KV cache，并丢弃或卸载复用价值较低的 suffix KV cache。

在理解 Hybrid Prefilling 和 Suffix KV Cache Offloading 的关系时，我的结论是：Hybrid Prefilling 是计算方式上的改造，Suffix KV Cache Offloading 是缓存管理策略。前者通过让 attention 层不切块、非 attention 层切块，降低 MLP 中间张量显存峰值，并使系统可以在一次完整 prefill 中安全丢弃或卸载 suffix KV cache。没有 hybrid prefilling，普通 chunked prefilling 可能需要反复使用前面 chunk 的 K/V，suffix cache 的丢弃/offload 就不容易做得高效。

最后，7.1 中的 routing 也帮助我理解了实验设置。对于非并行方案，系统在每张 GPU 上跑一个完整 inference engine 实例，然后按 user-id 路由：同一个用户的请求固定到同一个实例，以提高 prefix cache 命中率；不同用户通过 round-robin 分配以平衡负载。相比之下，tensor parallel 或 pipeline parallel 是一个模型实例跨多张 GPU，因此 KV cache 和模型参数被分摊到多张 GPU 上，有更多 prefix KV cache 空间，但代价是通信和同步开销。

## 我的理解

这篇论文的重要性在于，它指出 LLM serving 系统不应该只围绕生成式聊天 workload 设计。推荐、审核、分类、embedding 等应用中的 LLM 调用，经常只需要一次 prefill 和一个 token 的概率分布。这类 workload 的系统瓶颈和普通生成式 workload 不同，因此需要不同的显存管理和调度策略。

PrefillOnly 的设计逻辑很清晰：

- 既然没有多轮 decode，就不必为当前请求保留完整 KV cache；
- 既然输入很长，就必须控制 MLP 中间张量和 KV cache 两类显存开销；
- 既然输出长度固定，就可以进行更激进的 JCT-aware scheduling；
- 既然很多请求共享前缀，就不能简单丢弃所有 KV cache，而要保留最有复用价值的 prefix。

整体来看，PrefillOnly 是一篇典型的系统论文：它从新 workload 的特性出发，重新审视已有系统假设，并围绕显存、缓存和调度三个方面做专门优化。

# MCM-GPU: Multi-Chip-Module GPUs for Continued Performance Scalability

## 基本信息

- 论文：MCM-GPU: Multi-Chip-Module GPUs for Continued Performance Scalability
- 作者：Akhil Arunkumar, Evgeny Bolotin, Benjamin Cho, Ugljesa Milic, Eiman Ebrahimi, Oreste Villa, Aamer Jaleel, Carole-Jean Wu, David Nellans
- 会议：ISCA 2017
- 主题：GPU architecture、multi-chip module、chiplet GPU、NUMA、on-package interconnect、CTA scheduling、cache hierarchy

## 背景与问题

传统 GPU 性能增长很大程度上依赖晶体管缩放：随着工艺进步，单个 GPU die 可以集成更多 Streaming Multiprocessors（SM）、更大 cache 和更高内存带宽。但论文指出，这条路径会遇到几个限制：

- Moore's Law 放缓后，单位面积可集成晶体管数不再快速增长；
- 光刻 reticle size 限制了单个 die 的最大面积；
- 大 die 良率低，制造成本高；
- 即使应用有足够并行度，单芯片 GPU 也无法无限增加 SM 数量。

multi-GPU 是一种自然替代方案，但它也有明显问题。多个完整 GPU 通常通过板级或系统级互连协作，带宽更低、延迟更高、能耗更大，并且常常需要程序员或 runtime 显式处理任务划分、数据放置、同步和负载均衡。

本文提出的核心问题是：**能否在不制造超大 monolithic GPU die 的情况下，继续扩展单个逻辑 GPU 的性能？**

## 核心思想

论文提出 *MCM-GPU*，即把多个 GPU Module（GPM）集成到同一个 package 中，通过封装内互连连接，并对 OS 和程序员暴露成一个逻辑 GPU。

它的位置介于 monolithic GPU 和 multi-GPU 之间：

- monolithic GPU：所有计算和内存系统都在一个大 die 上；
- multi-GPU：多个完整 GPU 在 board/system 层级互连；
- MCM-GPU：多个较小 GPM/chiplet 在同一个 package 内互连，共同组成一个逻辑 GPU。

论文评估的主要设计是一个 256 SM 的 MCM-GPU，由 4 个 64 SM 的 GPM 组成。每个 GPM 包含一部分 SM、L1 cache、crossbar、L2 slice 和本地 DRAM partition。所有 GPM 的资源共同形成一个统一地址空间和一个逻辑 GPU。

## MCM-GPU 的组织方式

MCM-GPU 中的 L2 cache 容易引起混淆。更准确的理解是：

> L2 是逻辑上 shared、物理上 distributed/sliced 的。

也就是说，总 L2 由多个 GPM 内的 L2 slice 组成。所有 SM 都可以访问任意 L2 slice，但每个 physical address 只归属于一个 home GPM 的 L2 slice 和 DRAM partition。

例如：

```text
GPM0 的 SM 访问地址 A

如果 A 映射到 GPM0：
    GPM0 SM -> GPM0 L2 slice -> GPM0 DRAM

如果 A 映射到 GPM2：
    GPM0 SM -> inter-GPM link -> GPM2 L2 slice -> GPM2 DRAM
```

因此，baseline MCM-GPU 中每个 L2 slice 只缓存本地 DRAM partition 对应的数据。同一条 cache line 不会同时存在于多个 L2 slice 中，所以 L2 slice 之间不需要复杂硬件一致性协议。

## On-Package Bandwidth Considerations

这一节的核心问题是：**GPM 之间的封装内互连带宽要多大，才能不拖累整体 GPU 性能？**

论文用 4-GPM 系统进行估算。设每个 GPM 的本地 DRAM partition 带宽为 `b`，那么总 DRAM 带宽为 `4b`。在论文的配置中：

```text
总 DRAM bandwidth = 3 TB/s
每个 GPM 本地 DRAM bandwidth = b = 768 GB/s
```

如果地址在所有 GPM 的 memory partitions 上细粒度均匀分布，那么一个 GPM 的访存请求中，只有一部分访问本地 partition，很多请求会访问远端 partition。为了让所有 GPM 都能充分利用总 DRAM 带宽，inter-GPM bandwidth 理想上需要接近总 DRAM bandwidth，也就是约 `3 TB/s`。

但是这样的封装内互连成本、布线复杂度和功耗都较高。论文基于 NVIDIA 的 Ground-Referenced Signaling（GRS）技术，认为 `768 GB/s` 是更现实、低成本、低功耗的设计点。GRS 单线速率约为 `20 Gb/s`，因此 `768 GB/s` 级别带宽需要数百条并行高速信号线在 package 内连接相邻 GPM。

这也解释了后文优化的动机：baseline 选择的 `768 GB/s` inter-GPM link 低于理想需求，所以必须通过架构机制减少跨 GPM 流量。

## 三项优化机制

论文提出三个 locality-aware 优化，目标都是减少 remote access 和 inter-GPM bandwidth pressure。

### Remote-Only L1.5 Cache

L1.5 cache 是每个 GPM 内新增的一层 GPM-side cache，位于 SM 私有 L1 和远端 L2/DRAM 之间。它主要缓存本 GPM 访问过的远端数据。

baseline 中，如果 GPM0 访问 GPM2 的数据，后续再次访问时仍可能需要跨 GPM。加入 L1.5 后，第一次远端访问会把数据带回 GPM0 的 L1.5，后续访问可以在本地命中，从而减少 inter-GPM traffic 和远端访问延迟。

论文评估了 6 种 L1.5 配置：

| 配置 | 含义 |
|---|---|
| `8 MB L1.5` | 总 L1.5 为 8 MB，同时缓存本地和远端数据 |
| `8 MB Remote-Only L1.5` | 总 L1.5 为 8 MB，只缓存远端数据 |
| `16 MB L1.5` | 总 L1.5 为 16 MB，同时缓存本地和远端数据 |
| `16 MB Remote-Only L1.5` | 总 L1.5 为 16 MB，只缓存远端数据 |
| `32 MB L1.5` | 总 L1.5 为 32 MB，同时缓存本地和远端数据，额外增加 cache 容量 |
| `32 MB Remote-Only L1.5` | 总 L1.5 为 32 MB，只缓存远端数据，额外增加 cache 容量 |

其中 `8 MB` 和 `16 MB` 是 iso-transistor 配置，主要通过把原本 L2 的一部分容量挪到 L1.5 实现。`32 MB` 则额外增加 cache 容量，不是公平面积比较。

实验显示，remote-only 策略更符合 L1.5 的目标，因为本地数据已经由本地 L2 处理，L1.5 应优先保留代价更高的远端数据。单独使用 L1.5 时，`16 MB Remote-Only L1.5` 是两个 iso-transistor 配置中效果最好的设计。

### Distributed CTA Scheduling

CTA（Cooperative Thread Array）基本等同于 CUDA 中的 thread block。CTA 是 GPU 调度到 SM 上执行的基本单位。

baseline 的 centralized CTA scheduling 会根据 SM 可用情况全局分配 CTA，可能导致相邻 CTA 被分散到不同 GPM 中。例如：

```text
CTA0 -> GPM0
CTA1 -> GPM1
CTA2 -> GPM2
CTA3 -> GPM3
```

但相邻 CTA 往往访问相邻或重叠数据。如果它们被分散到不同 GPM，就会破坏 cache locality 并增加跨 GPM 访问。

Distributed CTA scheduling 将连续 CTA 分组分配到同一个 GPM，例如：

```text
CTA0, CTA1, CTA2, CTA3 -> GPM0
CTA4, CTA5, CTA6, CTA7 -> GPM1
```

这样可以提高 L1.5 cache 命中率，并减少 inter-GPM traffic。论文也指出，这种固定粗粒度分组可能造成负载不均，因此未来可以考虑动态选择 CTA group size。

### First-Touch Page Placement

First-touch（FT）策略用于页面放置。规则是：一个 memory page 第一次被哪个 GPM 访问，就被映射到哪个 GPM 的 local memory partition。

例如：

```text
CTA-X 在 GPM0 上第一次访问 page P0
=> P0 被放到 GPM0 的 memory partition

之后 GPM1 访问 P0
=> P0 仍在 GPM0
=> GPM1 需要远程访问
```

因此，FT 是 first-touch placement，不是动态 page migration。页面第一次放置后通常不再移动。

FT 和 distributed CTA scheduling 有很强的协同作用。很多 GPU 应用会反复 launch 相同 kernel，具有相同 index 的 CTA 在多次 kernel invocation 中可能访问相同页面。如果 distributed scheduling 让这些 CTA 稳定落在同一个 GPM，FT 就能让相关页面也稳定放在该 GPM 的本地 memory partition，从而跨 kernel 保持局部性。

## 实验设置与结果

论文使用 NVIDIA 内部模拟器评估一个类似 Pascal、但规模扩展到 256 SM 的 GPU。baseline 配置包括：

- 4 个 GPM；
- 总共 256 SM；
- 每个 SM 128 KB L1 data cache；
- 总 L2 cache 16 MB；
- inter-GPM ring interconnect，每条 link 768 GB/s，每 hop 32 cycles；
- 总 DRAM bandwidth 3 TB/s；
- DRAM latency 100 ns。

benchmark 共 48 个，来自 CORAL、Lonestar、Rodinia 和 NVIDIA 内部 CUDA workloads。论文将 workload 分为 high-parallelism 和 limited-parallelism，并进一步把 high-parallelism 分为 memory-intensive 和 compute-intensive。

主要结果：

- baseline MCM-GPU 在 inter-GPM bandwidth 低于 3 TB/s 时会明显受 NUMA 效应影响，memory-intensive workload 最敏感。
- 单独加入 remote-only L1.5 cache 可带来约 5.2% 平均性能提升，并显著降低 inter-GPM bandwidth。
- L1.5 cache + distributed CTA scheduling 对 memory-intensive workloads 平均提升约 23.4%。
- 加入 FT 后，最佳组合变为 `8 MB Remote-Only L1.5 + 8 MB L2 + Distributed Scheduling + First Touch`，因为 FT 使很多访问变成本地访问，此时保留足够 L2 容量更重要。
- 三项机制组合后，相比 baseline MCM-GPU 平均性能提升 22.8%，inter-GPM bandwidth 降低 5 倍。
- 优化后的 MCM-GPU 比论文假设的最大可制造 128 SM monolithic GPU 快 45.5%。
- 优化后的 MCM-GPU 比同等资源 multi-GPU 快 26.8%，并且距离不可制造的理想 256 SM monolithic GPU 性能差距在 10% 以内。

## 局限性

这篇论文也有一些限制：

- 结果基于 NVIDIA 内部模拟器和前瞻性硬件假设，不是实物芯片测量结果。
- `768 GB/s` inter-GPM bandwidth 基于 GRS 技术能力估算，但论文没有制造真实 MCM-GPU 验证。
- inter-GPM topology 主要采用 ring/mesh 的高层抽象，没有深入探索拓扑、routing 和拥塞控制。
- distributed CTA scheduling 的粒度固定，可能导致负载不均。
- FT page placement 不包含动态页面迁移，因此如果 first touch 与后续主要访问者不一致，仍可能产生远端访问。
- L1.5 cache 采用 write-through 以适配 GPU 软件一致性模型，这可能对写密集 workload 产生额外 DRAM traffic。

## 阅读过程中的问题与理解

我一开始主要困惑在 chiplet、MCM 和 multi-GPU 的关系。后来理清为：chiplet 是组成单位，即较小的 die；MCM 是把多个 die/chiplet 放进同一个 package 的封装方式；multi-GPU 则是多个完整 GPU 在板级或系统级互连。MCM-GPU 的目标不是暴露多个 GPU，而是把多个 GPM 组合成一个逻辑 GPU。

另一个关键概念是 package。这里的 package 不是软件包，而是芯片封装层级，介于单个 die 和 PCB board 之间。MCM-GPU 利用 package-level interconnect，让 GPM 间通信比 board-level multi-GPU 更近、更宽、更省电。

我也进一步澄清了 L2 cache 的组织方式。Section 4 中说 shared L2，是指 L2 对所有 SM 逻辑共享；但物理实现上，L2 被切成多个 slice 分布在各个 GPM 内。每个地址只对应一个 home L2 slice 和 DRAM partition，所以 L2 是 distributed shared cache，而不是一个集中式 L2。

L1.5 cache 的角色也需要和 L2 区分。L2 是 memory-side cache，负责某个 GPM 本地 DRAM partition 的数据；L1.5 是 GPM-side cache，主要缓存本 GPM 曾经访问过的远端数据。它不是全局共享 cache，而是减少远端访问的局部副本缓存。

对于 on-package bandwidth，我的理解是：论文不是说 `768 GB/s` 已经在产品中实现，而是把它作为基于 GRS 技术趋势的可行设计点。理论上，为了完全利用 3 TB/s 总 DRAM bandwidth，inter-GPM bandwidth 需要接近 3 TB/s；但这样的互连太贵。因此作者选择较现实的 768 GB/s，然后用 cache、CTA scheduling 和 page placement 降低实际跨 GPM 流量。

## 我的理解

这篇论文的重要性在于，它较早地把 chiplet/MCM 思路系统性引入 GPU 性能扩展问题。它的核心洞察不是简单地“把多个小 GPU 拼起来”，而是指出这样的系统天然会成为 NUMA GPU，因此必须围绕 locality 重新设计 cache、调度和页面放置。

MCM-GPU 的设计逻辑可以概括为：

- 用多个较小 GPM 避免超大 monolithic die 的制造限制；
- 用 package-level interconnect 避免 board-level multi-GPU 的高延迟、高能耗和编程复杂度；
- 用统一逻辑 GPU 把复杂性尽量隐藏在硬件和驱动中；
- 用 L1.5 cache、distributed CTA scheduling 和 FT page placement 缓解 NUMA 带来的远端访问开销。

整体来看，MCM-GPU 是一篇典型的体系结构论文：它先指出未来单芯片扩展的物理限制，再提出一个新的封装级组织方式，最后围绕该组织方式产生的新瓶颈设计配套机制。它对后来的 chiplet GPU 和先进封装 GPU 设计有很强的启发意义。
